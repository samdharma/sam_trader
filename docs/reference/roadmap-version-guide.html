5. Understanding Git Branches

If you're new to Git, think of branches like parallel universes of your code. Each branch is a complete copy of the project at a point in time. Changes on one branch do not affect other branches until you explicitly merge them.

Three branches, three purposes

BranchPurposeWhat lives thereDeployed where
master Production — stable, tested v2.0 (current) Mac Mini
v2.5-dev Development — new features v2.5 work in progress Nowhere yet (testing only)
(future) v3.0 development branch Futu broker + advanced dashboard Nowhere yet

Why not work directly on master?

Master is your "live" code — the version running on the Mac Mini right now. If you make changes directly on master and something breaks, you have no easy way to go back to the working version. A development branch gives you a safe sandbox where you can experiment, fail, and retry without risk.

Merging: bringing work together

When v2.5 is complete and tested, you merge the v2.5-dev branch into master. This takes all the new features and applies them to the production code. After merging, you tag the result as v2.5.0 so you can always go back to that exact version if needed.

Hotfixes: what if master has a bug while v2.5 is being built?

This is where branches shine. You can:

  1. Switch to master: git checkout master
  2. Fix the bug, commit
  3. Deploy the fix to the Mac Mini
  4. Switch back to v2.5-dev: git checkout v2.5-dev
  5. Pull the fix in: git merge master

The v2.5 branch gets the fix without disrupting ongoing development. Master gets the fix without waiting for v2.5 to be ready.

Tags: snapshot of a specific version

A tag is like a bookmark. When you merge v2.5 into master, you create a tag v2.5.0. This means you (or any operator) can always go back to exactly that point in time: 

git checkout v2.5.0          # switch to the exact v2.5 release
git checkout v2.0            # switch to the v2.0 release tag

6. Day-to-Day: Working with Versions

6.1 Checking what version you're on

cd /path/to/csam_trader

# See which branch you're on
git branch
# The branch with * is your current branch

# See recent commits
git log --oneline -5

# See all tags (version markers)
git tag

6.2 Staying on v2.0 (current production)

If you just want to keep the Mac Mini running and apply occasional fixes:

cd /path/to/csam_trader

# Ensure you're on master
git checkout master

# Pull latest fixes
git pull --rebase

# Apply to running system
./deploy.sh --restart

6.3 Switching to v2.5 when it's ready

When v2.5 development is complete and merged to master:

cd /path/to/csam_trader

# Get the latest master (now includes v2.5)
git checkout master
git pull --rebase

# Verify you have v2.5
git tag | grep v2.5

# Deploy the update
./deploy.sh --restart --force

6.4 Working on v2.5 development (ralph loop)

cd /path/to/csam_trader

# Switch to the development branch
git checkout v2.5-dev

# Start the ralph loop (will pick up v2.5 tickets)
bash scripts/ralph/run_ralph_loop.sh

# To switch back to master (for production fixes):
git checkout master
