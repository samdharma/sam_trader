# EPIC-5: Pre-Market Orchestration & Handoff

> **EPIC ID:** EPIC-5  
> **Status:** Active  
> **Phase:** Pre-Market (09:00–09:25 ET / HKT)  
> **Depends On:** EPIC-1 (Gap Discovery), EPIC-2 (AI Analysis), EPIC-3 (Risk & Sizing), EPIC-4 (Regime Detection)  
> **Out of Scope:** Live trade execution, intraday monitoring, end-of-day close

---

## 1. EPIC Description

As a **trading system operator**, I want a single orchestrated pipeline that aggregates all pre-market activities — gap scans, AI analysis, risk validation, and regime detection — into a unified, validated handoff artifact, so that market operations can begin the trading day with confidence and full situational awareness.

### 1.1 Business Value

- **Operational Confidence:** The sanity check validates that all critical systems are healthy before market open. A failed sanity check blocks trading, preventing losses from known issues.
- **Single Source of Truth:** The Execution Bundle is the canonical handoff artifact. No ambiguity, no missing data, no conflicting versions.
- **Auditability:** Every stage of the pipeline is logged, timestamped, and traceable. Operators can reconstruct exactly what happened and why.
- **Automation:** The entire pipeline runs on a fixed schedule without human intervention, yet produces human-readable reports for operator review.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Aggregation of multiple gap scans
  ✓ Deduplication and trend filtering
  ✓ Orchestration of AI analysis across candidates
  ✓ System health validation (sanity check)
  ✓ Execution bundle creation and persistence
  ✓ Daily readiness reporting
  ✓ Error handling and alerting
  ✓ Operator notification

OUT OF SCOPE:
  ✗ Order submission or execution
  ✗ Intraday pipeline re-runs
  ✗ Post-market analysis or reporting
  ✗ Manual trade entry via UI
```

## 2. Features

### Feature-5.1: Pre-Market Pipeline Orchestrator

#### Description
The central orchestration module that runs the pre-market pipeline. It discovers all gap scans from the current trading day, deduplicates candidates, filters fading gaps, runs AI analysis, invokes the portfolio manager, and assembles the master candidate list.

#### User Stories

##### US-5.1.1: Scan Discovery and Aggregation
> **As a** trading system,  
> **I want** to discover all gap scans run during the current trading day and aggregate them into a single master list,  
> **So that** no candidate is missed and duplicate symbols are handled consistently.

**Acceptance Criteria:**
- [ ] The pipeline discovers all scan result files for the current trading date.
- [ ] It supports multiple scan types: broad universe scans, enhanced scans, and HK scans.
- [ ] Candidates from the same symbol appearing in multiple scans are deduplicated.
- [ ] Deduplication keeps the most recent scan data for each symbol.
- [ ] The pipeline reports: total scans discovered, total candidates before deduplication, and candidates after deduplication.
- [ ] If no scans are found for the current date, the pipeline logs a warning and attempts to use the most recent available scans (within 24 hours).

**Functional Validation:**
```
Input:  Scan #1 (04:30): AAPL, TSLA, MSFT
        Scan #2 (08:30): AAPL (updated), TSLA (updated), NVDA (new)
Output: Master list: AAPL (from Scan #2), TSLA (from Scan #2),
        MSFT (from Scan #1), NVDA (from Scan #2)

Validation:
  1. 4 unique symbols after deduplication.
  2. AAPL and TSLA use Scan #2 data (most recent).
  3. MSFT retained from Scan #1.
  4. NVDA added from Scan #2.
  5. Pipeline reports: 2 scans, 6 candidates before dedup, 4 after.
```

##### US-5.1.2: Trend-Down Filtering
> **As a** trading system,  
> **I want** to filter out symbols whose gaps are fading between scan passes,  
> **So that** the system does not chase opportunities that are already deteriorating.

**Acceptance Criteria:**
- [ ] For symbols appearing in multiple scans, the pipeline calculates gap decay: `(gap_scan1 - gap_scan2) / gap_scan1 × 100`.
- [ ] If gap decay exceeds a configurable threshold (default: 20%), the symbol is excluded.
- [ ] Excluded symbols are logged with the reason "TREND_DOWN" and the decay percentage.
- [ ] Symbols with gap decay ≤ threshold are retained.
- [ ] Symbols appearing in only one scan are retained (no decay data).

**Functional Validation:**
```
Input:  Scan #1: TSLA gap=8.0%; Scan #2: TSLA gap=5.5%; threshold=20%
Output: TSLA excluded

Validation:
  1. Decay = (8.0 - 5.5) / 8.0 × 100 = 31.25%.
  2. 31.25% > 20% → excluded.
  3. Log: "Excluded TSLA: TREND_DOWN (decay: 31.25%)".

Input:  Scan #1: AAPL gap=4.0%; Scan #2: AAPL gap=3.5%; threshold=20%
Output: AAPL retained

Validation:
  1. Decay = (4.0 - 3.5) / 4.0 × 100 = 12.5%.
  2. 12.5% ≤ 20% → retained.
```

##### US-5.1.3: AI Analysis Orchestration
> **As a** trading system,  
> **I want** to run AI analysis on the master candidate list,  
> **So that** every candidate receives a scored recommendation before portfolio construction.

**Acceptance Criteria:**
- [ ] AI analysis runs on all candidates in the master list.
- [ ] Analysis is parallelized where possible to complete within the pipeline timeout.
- [ ] Failed analyses (e.g., API timeout) are retried once with a secondary provider.
- [ ] If all providers fail for a candidate, rule-based analysis is applied.
- [ ] The pipeline reports: total candidates analyzed, AI-success count, fallback count, and failure count.
- [ ] All AI recommendations are validated to ensure they contain required fields.

**Functional Validation:**
```
Input:  Master list: 20 candidates
Output: 20 AI recommendations

Validation:
  1. All 20 candidates have recommendations.
  2. At least 15 are from AI providers (not fallback).
  3. All recommendations have: symbol, recommendation, confidence, total_score.
  4. Pipeline reports success/failure counts accurately.
```

##### US-5.1.4: Portfolio Manager Invocation
> **As a** trading system,  
> **I want** to invoke the portfolio manager after AI analysis completes,  
> **So that** the final portfolio recommendation reflects both new opportunities and existing positions.

**Acceptance Criteria:**
- [ ] The portfolio manager is invoked with:
  - AI recommendations (score ≥ 60)
  - Current portfolio positions and market values
  - Cash balance
  - Risk constraints from configuration
  - Current regime from EPIC-4
- [ ] The portfolio manager produces a PortfolioRecommendation within 60 seconds.
- [ ] If the portfolio manager fails, the pipeline falls back to a simple equal-weight allocation among top-scoring candidates.
- [ ] The portfolio recommendation is included in the pipeline output.

**Functional Validation:**
```
Input:  5 AI recommendations with scores 85, 78, 72, 65, 60
        Current positions: AAPL=$10K
        NAV=$100K, cash=$20K
Output: Portfolio recommendation

Validation:
  1. Portfolio manager completes within 60s.
  2. Recommendation respects max positions, max position %, max sector %, min cash %.
  3. Rebalancing orders are generated if current portfolio deviates from target.
  4. If portfolio manager fails, fallback produces equal-weight top-3 allocation.
```

### Feature-5.2: Pre-Market Sanity Check

#### Description
A comprehensive system health validation that runs at 09:15 ET (before market open) to verify that all critical components are functional. It checks kill switch state, broker connectivity, data freshness, AI pipeline health, risk manager load, and more.

#### User Stories

##### US-5.2.1: System Health Validation
> **As a** trading system operator,  
> **I want** to run a comprehensive sanity check before market open,  
> **So that** I can be confident the system is safe to trade or know exactly what needs fixing.

**Acceptance Criteria:**
- [ ] The sanity check includes at minimum the following validations:
  1. **Kill Switch:** Status is INACTIVE (with confirmation code check for live trading).
  2. **Broker Connectivity:** Broker API is reachable and responds to a health probe.
  3. **Data Client:** Market data source is reachable (can retrieve a benchmark symbol like SPY).
  4. **Stuck Orders:** No PENDING_NEW orders remain from prior sessions.
  5. **Database:** Primary storage is reachable and writable (test insert + rollback).
  6. **AI Pipeline:** AI analyzer initializes without error.
  7. **Position Sizer / Risk Manager:** Both load without error and produce valid outputs on a test input.
  8. **Last Gap Scan:** Most recent gap scan is within 24 hours.
  9. **Regime Model:** HMM regime model exists and is loadable.
  10. **ML Model Health:** Machine learning models (e.g., gap predictor) are present and fresh.
  11. **Position Reconciliation:** Broker positions match local ledger within tolerance.
  12. **Stale Positions:** No day positions open for > 3 market days.
  13. **System Config:** All required configuration values are present and valid.
- [ ] Each check produces a PASS/FAIL result with a message and optional details.
- [ ] Failed checks include a remediation hint (actionable instruction for the operator).
- [ ] The sanity check completes within 2 minutes.
- [ ] Any FAIL result triggers a CRITICAL alert to the operator.

**Functional Validation:**
```
Input:  All systems healthy
Output: Sanity report with all checks PASS

Validation:
  1. All 13 checks report PASS.
  2. Total duration < 120 seconds.
  3. No alerts sent.

Input:  Database unreachable, last gap scan > 24h old
Output: Sanity report with 2 FAILs

Validation:
  1. Database check FAIL with message and remediation hint.
  2. Gap scan check FAIL with remediation hint.
  3. CRITICAL alert sent to operator.
  4. Report suggests: "Do not proceed with trading until resolved."
```

##### US-5.2.2: Readiness Reporting
> **As a** trading system operator,  
> **I want** a human-readable daily readiness report,  
> **So that** I can review system health, scan results, and trading readiness at a glance.

**Acceptance Criteria:**
- [ ] The report includes:
  - System health summary (PASS/FAIL per check)
  - Last gap scan summary (candidate count, top 5 symbols)
  - AI recommendation summary (count per grade: STRONG_BUY, BUY, etc.)
  - Portfolio status (open positions, NAV, cash, heat)
  - Regime classification with confidence
  - Kill switch and circuit breaker status
  - Today's trading schedule and expected actions
- [ ] The report is generated in HTML format for operator review.
- [ ] The report is persisted with the trading date in the filename.
- [ ] The report is available within 5 minutes of the sanity check completion.

**Functional Validation:**
```
Input:  Typical pre-market morning
Output: readiness_report_YYYY-MM-DD.html

Validation:
  1. File exists and is valid HTML.
  2. Contains all required sections.
  3. Top 5 symbols are correctly ranked by gap % or AI score.
  4. NAV and cash figures are accurate to within $1.
  5. Report generation time < 300 seconds.
```

### Feature-5.3: Execution Bundle Creator

#### Description
The final stage of pre-market that assembles all outputs — gap candidates, AI recommendations, portfolio recommendation, regime state, and safety state — into a single canonical artifact for handoff to market operations.

#### User Stories

##### US-5.3.1: Bundle Assembly
> **As a** trading system,  
> **I want** to assemble all pre-market outputs into a single execution bundle,  
> **So that** market operations have one canonical source of truth for the trading day.

**Acceptance Criteria:**
- [ ] The bundle contains all six sections defined in the functional overview:
  1. Scan Metadata (timestamp, date, trace ID)
  2. Gap Candidates (full list with all fields)
  3. AI Recommendations (full list with scores and parameters)
  4. Portfolio Recommendation (targets, rebalancing orders, cash target)
  5. Regime State (regime, confidence, stability)
  6. Safety State (kill switch, circuit breaker)
- [ ] The bundle format is structured and machine-readable.
- [ ] The bundle is persisted with a filename containing the trading date.
- [ ] A symlink or reference to the latest bundle is maintained for easy retrieval.
- [ ] The bundle size is reasonable (< 10 MB for 50 candidates).

**Functional Validation:**
```
Input:  Pipeline results: 20 candidates, 15 AI recommendations,
        1 portfolio recommendation, regime=LOW_VOL, kill_switch=INACTIVE
Output: execution_bundle_YYYY-MM-DD_HHMMSS.json

Validation:
  1. File exists and is valid structured data.
  2. All 6 sections are present and non-empty.
  3. trace_id is present and is a valid UUID or trace string.
  4. All symbols are valid ticker formats (1–5 uppercase characters, no spaces).
  5. File size < 10 MB.
  6. Latest bundle reference is updated.
```

##### US-5.3.2: Bundle Validation
> **As a** trading system,  
> **I want** to validate the execution bundle before handoff,  
> **So that** corrupted or incomplete bundles do not reach market operations.

**Acceptance Criteria:**
- [ ] Bundle validation checks:
  1. All required sections are present.
  2. All symbols are valid ticker formats.
  3. All prices are positive numbers.
  4. All percentages are within valid ranges (0%–100%).
  5. The trace ID is present and non-empty.
  6. The scan timestamp is within the last 4 hours.
  7. At least one candidate or one portfolio rebalancing order is present (non-empty trading day).
- [ ] If validation fails, the pipeline logs detailed errors and aborts handoff.
- [ ] If validation passes, the bundle is marked as "validated" with a timestamp.

**Functional Validation:**
```
Input:  Bundle with missing AI recommendations section
Output: Validation FAILED

Validation:
  1. Missing section detected.
  2. Error log: "Bundle validation failed: missing section 'ai_recommendations'".
  3. Handoff aborted.

Input:  Complete, valid bundle
Output: Validation PASSED

Validation:
  1. All checks pass.
  2. Bundle marked with "validated_at" timestamp.
  3. Handoff proceeds.
```

##### US-5.3.3: Bundle-to-Signals Conversion
> **As a** market operations system,  
> **I want** to convert the execution bundle into standardized trade signals,  
> **So that** the execution engine can route them without re-parsing individual pipeline outputs.

**Acceptance Criteria:**
- [ ] Portfolio rebalancing orders are converted to signals first (preserving priority).
- [ ] AI recommendations graded STRONG_BUY or BUY are converted to ENTRY signals.
- [ ] Each signal includes: symbol, direction (LONG/SHORT), entry price, stop price, target price, position size (dollar amount or shares), strategy name, confidence, regime, and trace ID.
- [ ] Signals for blacklisted or invalid symbols are filtered out.
- [ ] The signal list is ordered: rebalancing first, then entries by confidence descending.

**Functional Validation:**
```
Input:  Bundle with: rebalance order (sell TSLA), STRONG_BUY AAPL, BUY MSFT
Output: Signal list: [SELL TSLA, BUY AAPL, BUY MSFT]

Validation:
  1. Rebalance signal appears first.
  2. AAPL (STRONG_BUY) appears before MSFT (BUY).
  3. Each signal has: symbol, direction, entry, stop, target, size, trace_id.
  4. Invalid symbols are excluded.
```

### Feature-5.4: Daily Readiness Report

#### Description
An executive summary generated early in the morning (05:15 ET) that gives the operator a quick view of system health before any trading activity begins. It serves as the first go/no-go checkpoint of the day.

#### User Stories

##### US-5.4.1: Early Morning System Check
> **As a** trading system operator,  
> **I want** to receive a daily readiness report at 05:15 ET,  
> **So that** I can identify and fix issues before the pre-market pipeline begins at 09:00 ET.

**Acceptance Criteria:**
- [ ] The readiness report checks:
  1. Python runtime and virtual environment
  2. Environment configuration file present and readable
  3. Primary storage reachable and functional
  4. Broker API connection (paper or live, as configured)
  5. Kill switch state (must be INACTIVE)
- [ ] Each check is graded: PASS (green), WARNING (yellow), FAIL (red).
- [ ] Any FAIL result triggers an immediate Telegram alert.
- [ ] The report is generated in HTML and persisted to `reports/YYYY-MM-DD/`.
- [ ] The report completes within 2 minutes.

**Functional Validation:**
```
Input:  All systems healthy at 05:15 ET
Output: readiness_report_YYYY-MM-DD.html with all PASS

Validation:
  1. All 5 checks report PASS.
  2. HTML file is valid and readable.
  3. No alerts sent.
  4. Duration < 120 seconds.

Input:  Broker API unreachable at 05:15 ET
Output: readiness report with broker check FAIL

Validation:
  1. Broker check reports FAIL (red).
  2. CRITICAL alert sent immediately.
  3. Report includes remediation hint.
```

## 3. Functional Requirements

### FR-5.1: Pipeline Timeout
The pre-market pipeline must complete within 10 minutes of initiation. If it exceeds this timeout, it must abort with a CRITICAL alert and partial results must not be handed off.

### FR-5.2: Atomic Handoff
The execution bundle must be fully written and validated before it is marked as available to market operations. Partial or corrupted bundles must not be consumed.

### FR-5.3: Trace Propagation
The same trace identifier must flow through all pipeline stages: scan discovery → AI analysis → portfolio manager → bundle creation → sanity check. This enables end-to-end debugging.

### FR-5.4: Idempotency
Running the pipeline twice with identical inputs must produce identical outputs (within timestamp differences). Non-deterministic components must be seeded or bounded.

### FR-5.5: Alert Escalation
- WARNING-level issues (e.g., single scan failure, one AI timeout) are logged but do not block trading.
- CRITICAL-level issues (e.g., sanity check failure, bundle validation failure) block trading and alert the operator immediately.

### FR-5.6: Schedule Independence
The pipeline must be runnable on-demand by an operator, not just via the scheduled cron job. Manual runs must produce the same outputs as scheduled runs.

## 4. Acceptance Criteria (EPIC Level)

- [ ] **AC-5.1:** The pre-market pipeline completes successfully within 10 minutes on 95% of trading days.
- [ ] **AC-5.2:** The execution bundle contains all required sections and passes validation on every successful pipeline run.
- [ ] **AC-5.3:** The sanity check detects and reports any kill switch activation, blocking trading until resolved.
- [ ] **AC-5.4:** If a gap scan is missing, the pipeline uses the most recent available scan (within 24h) and logs a warning.
- [ ] **AC-5.5:** The daily readiness report is generated by 05:17 ET and is accessible via the reports directory.
- [ ] **AC-5.6:** Any CRITICAL failure in the pipeline or sanity check sends an alert within 60 seconds.
