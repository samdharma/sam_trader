# EPIC-6: Safety & Emergency Controls

> **EPIC ID:** EPIC-6  
> **Status:** Active  
> **Phase:** Pre-Market (checked at 09:15 ET sanity check; active throughout trading day)  
> **Out of Scope:** Intraday kill switch triggering (handled by live monitor), post-trade kill switch analysis, regulatory reporting

---

## 1. EPIC Description

As a **trading system operator**, I want robust safety and emergency controls that can halt or throttle trading immediately when critical thresholds are breached or when manual intervention is required, so that catastrophic losses are prevented and the system respects the operator's authority at all times.

### 1.1 Business Value

- **Catastrophic Loss Prevention:** The kill switch provides an immediate, irreversible halt to all trading activity when the system or market enters a dangerous state.
- **Operator Authority:** A manual kill switch ensures that a human operator can always override the system, even if all automated checks pass.
- **Graduated Response:** Circuit breakers provide a graduated response (reduce 50% → halt) rather than an all-or-nothing approach, giving the system room to recover.
- **Audit Trail:** Every activation, deactivation, and trigger is logged with full context for post-incident analysis.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Kill switch state management (active / inactive)
  ✓ Manual activation and deactivation
  ✓ Automatic activation triggers (circuit breaker, drawdown, error rate)
  ✓ Pre-market kill switch validation
  ✓ Kill switch confirmation code for live trading
  ✓ Circuit breaker state management (normal / reduced_50 / halted)
  ✓ Circuit breaker trigger evaluation
  ✓ Event logging and alerting

OUT OF SCOPE:
  ✗ Intraday real-time kill switch monitoring (handled by intraday monitor)
  ✗ Post-incident root cause analysis tools
  ✗ Regulatory filing or reporting
  ✗ Insurance or compliance workflows
```

## 2. Features

### Feature-6.1: Kill Switch & Emergency Halt

#### Description
A thread-safe emergency halt mechanism that immediately prevents all new trading activity when activated. It supports both manual operator activation and automatic activation based on predefined triggers.

#### User Stories

##### US-6.1.1: Manual Kill Switch Activation
> **As a** trading system operator,  
> **I want** to activate an emergency kill switch at any time,  
> **So that** I can immediately halt all trading when I judge the market or system to be unsafe.

**Acceptance Criteria:**
- [ ] The kill switch can be activated via a command-line interface.
- [ ] For live trading, activation requires a confirmation code (prevents accidental activation).
- [ ] Activation is immediate: all new order submissions are blocked within 1 second.
- [ ] Existing open positions are NOT automatically closed (operator decides separately).
- [ ] Activation is logged with: timestamp, reason (MANUAL), triggered_by (operator identity), equity at trigger, and hostname.
- [ ] A CRITICAL alert is sent to the operator upon activation.
- [ ] The kill switch state is persisted to disk so it survives process restarts.

**Functional Validation:**
```
Input:  Operator runs: activate kill-switch --reason "Market crash" --code "SECRET123"
Output: KillSwitchState(is_active=true)

Validation:
  1. Confirmation code is verified for live trading.
  2. State becomes ACTIVE within 1 second.
  3. New orders are rejected with reason "Kill switch active".
  4. Log entry contains timestamp, reason, operator, and equity.
  5. CRITICAL alert sent.
  6. State file on disk shows "is_active": true.
```

##### US-6.1.2: Manual Kill Switch Deactivation
> **As a** trading system operator,  
> **I want** to deactivate the kill switch after resolving the issue,  
> **So that** trading can resume under my explicit authorization.

**Acceptance Criteria:**
- [ ] Deactivation requires the same confirmation code as activation (for live trading).
- [ ] Deactivation is logged with: timestamp, reason, triggered_by, and hostname.
- [ ] After deactivation, a cooldown period (default: 60 seconds) must pass before new orders are accepted.
- [ ] During cooldown, orders are rejected with reason "Kill switch cooldown active".
- [ ] A notification is sent to the operator when the cooldown expires and trading resumes.

**Functional Validation:**
```
Input:  Operator runs: deactivate kill-switch --code "SECRET123"
Output: KillSwitchState(is_active=false, is_in_cooldown=true)

Validation:
  1. Confirmation code verified.
  2. State becomes INACTIVE.
  3. Cooldown flag is true.
  4. Orders rejected for 60 seconds with cooldown reason.
  5. After 60s, cooldown flag false, orders accepted.
  6. Notification sent when cooldown expires.
```

##### US-6.1.3: Automatic Kill Switch Activation
> **As a** risk manager,  
> **I want** the kill switch to activate automatically when critical thresholds are breached,  
> **So that** the system halts itself even if the operator is unavailable.

**Acceptance Criteria:**
- [ ] Automatic activation triggers include:
  - **CIRCUIT_BREAKER:** Daily PnL falls below the halt threshold.
  - **DRAWDOWN_LIMIT:** Account drawdown from peak exceeds configured limit.
  - **ERROR_RATE:** Error rate exceeds acceptable threshold (e.g., > 10% of operations failing).
  - **CONNECTIVITY:** Loss of connection to critical services (broker, data, storage) for > 5 minutes.
  - **MARKET_DATA:** Bad market data detected (e.g., prices = 0, impossible gaps).
  - **RISK_LIMIT:** Portfolio heat or single-position exposure exceeds absolute hard limits.
  - **SYSTEM_FAILURE:** Critical system component failure (e.g., execution engine crash).
- [ ] Each automatic trigger is logged with the trigger type, threshold, actual value, and timestamp.
- [ ] Automatic activation sends a CRITICAL alert with remediation hints.
- [ ] Automatic activations from test sources (regression_test, pytest, ci_pipeline) are automatically deactivated to prevent test artifacts from blocking production.

**Functional Validation:**
```
Input:  Daily PnL = -5.0%; daily_pnl_halt = -3.0%
Output: Kill switch ACTIVATED, reason=CIRCUIT_BREAKER

Validation:
  1. Trigger detected: daily_pnl (-5.0%) ≤ halt threshold (-3.0%).
  2. Kill switch activated automatically.
  3. Log: "Auto-activated: CIRCUIT_BREAKER (daily_pnl=-5.0%, threshold=-3.0%)".
  4. CRITICAL alert sent.

Input:  Trigger source = "pytest", daily_pnl = -5.0%
Output: Kill switch auto-activated, then auto-deactivated

Validation:
  1. Kill switch activates due to circuit breaker.
  2. Source "pytest" is in test trigger list.
  3. System auto-deactivates kill switch.
  4. Log: "Auto-deactivated test trigger: pytest".
```

##### US-6.1.4: Pre-Market Kill Switch Validation
> **As a** trading system,  
> **I want** to verify the kill switch state during the pre-market sanity check,  
> **So that** trading never begins while the kill switch is active.

**Acceptance Criteria:**
- [ ] The sanity check reads the kill switch state from disk.
- [ ] If the kill switch is ACTIVE, the sanity check FAILS.
- [ ] The failure message includes the activation reason, timestamp, and who triggered it.
- [ ] For live trading, the sanity check also verifies that a confirmation code is configured.
- [ ] If kill switch is INACTIVE, the check PASSES.

**Functional Validation:**
```
Input:  Kill switch state: ACTIVE, reason="MANUAL", triggered_by="operator"
Output: Sanity check FAIL

Validation:
  1. Kill switch check reports FAIL.
  2. Message: "Kill switch ACTIVE (MANUAL by operator at 2026-05-17T09:00:00Z)".
  3. Remediation hint: "Run: deactivate kill-switch to resume trading".
  4. Trading blocked.

Input:  Kill switch state: INACTIVE
Output: Sanity check PASS

Validation:
  1. Kill switch check reports PASS.
  2. Trading may proceed (other checks permitting).
```

### Feature-6.2: Circuit Breaker Management

#### Description
A graduated risk control mechanism that reduces or halts trading based on real-time P&L performance. It provides an automatic, rules-based throttle before the more drastic kill switch is needed.

#### User Stories

##### US-6.2.1: Circuit Breaker Evaluation
> **As a** trading system,  
> **I want** to evaluate the circuit breaker status before every trade,  
> **So that** trades are blocked or reduced when P&L deteriorates.

**Acceptance Criteria:**
- [ ] Circuit breaker evaluates three metrics:
  - **Daily PnL:** Percentage change from starting equity today.
  - **Weekly PnL:** Percentage change from starting equity this week.
  - **Peak Drawdown:** Percentage decline from the high water mark.
- [ ] State transitions:
  - NORMAL → REDUCED_50: When daily PnL ≤ daily_pnl_reduce threshold.
  - NORMAL → HALTED: When daily PnL ≤ daily_pnl_halt threshold.
  - REDUCED_50 → HALTED: When daily PnL ≤ daily_pnl_halt threshold.
  - REDUCED_50 → NORMAL: At start of next trading day.
  - HALTED → NORMAL: At start of next trading day (manual reset not required for circuit breaker).
- [ ] Evaluation completes within 100 milliseconds.
- [ ] The result includes: current daily PnL, weekly PnL, peak drawdown, current state, and whether each threshold is breached.

**Functional Validation:**
```
Input:  Starting equity = $100K; Current equity = $97.5K;
        daily_pnl_reduce = -2.0%; daily_pnl_halt = -3.0%
Output: Circuit breaker = REDUCED_50

Validation:
  1. Daily PnL = (97.5 - 100) / 100 = -2.5%.
  2. -2.5% ≤ -2.0% → REDUCED_50.
  3. -2.5% > -3.0% → HALTED not triggered.
  4. New position sizes are halved.

Input:  Current equity = $96.5K
Output: Circuit breaker = HALTED

Validation:
  1. Daily PnL = -3.5%.
  2. -3.5% ≤ -3.0% → HALTED.
  3. All new entries blocked.
```

##### US-6.2.2: Venue-Specific Circuit Breaker Thresholds
> **As a** dual-market trading system,  
> **I want** different circuit breaker thresholds for US and HK markets,  
> **So that** each market's unique volatility characteristics are respected.

**Acceptance Criteria:**
- [ ] US venue thresholds (defaults):
  - daily_pnl_reduce: -2.0%
  - daily_pnl_halt: -3.0%
- [ ] HK venue thresholds (defaults):
  - daily_pnl_reduce: -1.5%
  - daily_pnl_halt: -2.5%
- [ ] The risk manager applies the correct thresholds based on the trade's venue.
- [ ] Thresholds are configurable per venue without code changes.

**Functional Validation:**
```
Input:  US trade, daily_pnl = -2.2%
Output: REDUCED_50 (US threshold: -2.0%)

Validation:
  1. US threshold applied.
  2. -2.2% ≤ -2.0% → REDUCED_50.

Input:  HK trade, daily_pnl = -2.2%
Output: HALTED (HK threshold: -2.5% for halt, but -2.2% > -2.5%... wait,
        -2.2% ≤ -1.5% → REDUCED_50; -2.2% > -2.5% → not halted)

Validation:
  1. HK threshold applied.
  2. -2.2% ≤ -1.5% → REDUCED_50.
  3. -2.2% > -2.5% → not halted.
```

##### US-6.2.3: High Water Mark Tracking
> **As a** risk manager,  
> **I want** the system to track the peak account equity (high water mark),  
> **So that** drawdown is measured from the peak, not from an arbitrary starting point.

**Acceptance Criteria:**
- [ ] The high water mark is updated whenever account equity exceeds the previous high water mark.
- [ ] Drawdown is calculated as: `(current_equity - high_water_mark) / high_water_mark`.
- [ ] The high water mark persists across trading days.
- [ ] A separate daily starting equity is recorded at the beginning of each trading day for daily PnL calculation.

**Functional Validation:**
```
Input:  HWM = $105K; Current equity = $100K
Output: Drawdown = (100 - 105) / 105 = -4.76%

Validation:
  1. Drawdown correctly calculated from HWM.
  2. If equity rises to $106K, HWM updates to $106K.
  3. Drawdown resets to 0%.
```

## 3. Functional Requirements

### FR-6.1: Immediate Effect
Kill switch activation must take effect within 1 second. No new orders may be accepted after activation.

### FR-6.2: Persistence
Kill switch state must be persisted to disk and survive process restarts. A system restart must not inadvertently clear an active kill switch.

### FR-6.3: Confirmation Code
For live trading environments, kill switch deactivation must require a confirmation code. Paper trading may optionally skip confirmation for operator convenience.

### FR-6.4: Test Trigger Auto-Reset
Kill switch activations triggered by test frameworks (pytest, regression_test, ci_pipeline) must be automatically deactivated to prevent test artifacts from blocking production trading.

### FR-6.5: Circuit Breaker Daily Reset
Circuit breaker state must automatically reset to NORMAL at the start of each trading day. A circuit breaker triggered on Monday must not persist to Tuesday.

### FR-6.6: Alert Fatigue Prevention
Repeated kill switch or circuit breaker alerts within a 5-minute window must be deduplicated. The operator must not receive 50 identical alerts for the same condition.

### FR-6.7: Audit Completeness
Every kill switch event (activation, deactivation, auto-reset) must be logged with: timestamp, action, reason, triggered_by, equity_at_trigger, daily_pnl_at_trigger, session_pnl_at_trigger, hostname, and who (operator or system).

## 4. Acceptance Criteria (EPIC Level)

- [ ] **AC-6.1:** Activating the kill switch blocks all new orders within 1 second.
- [ ] **AC-6.2:** The kill switch state persists across process restarts.
- [ ] **AC-6.3:** Deactivating the kill switch requires a confirmation code in live trading mode.
- [ ] **AC-6.4:** A test-triggered kill switch is automatically deactivated within 5 seconds.
- [ ] **AC-6.5:** Circuit breaker transitions from NORMAL to REDUCED_50 when daily PnL crosses the reduce threshold.
- [ ] **AC-6.6:** Circuit breaker transitions from REDUCED_50 to HALTED when daily PnL crosses the halt threshold.
- [ ] **AC-6.7:** Circuit breaker resets to NORMAL at the start of the next trading day.
- [ ] **AC-6.8:** All kill switch and circuit breaker events are logged with full context and are auditable.
