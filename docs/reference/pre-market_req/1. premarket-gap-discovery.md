# EPIC-1: Pre-Market Gap Discovery & Scanning

> **EPIC ID:** EPIC-1  
> **Status:** Active  
> **Markets:** US Equities (primary), HK Equities (secondary)  
> **Phase:** Pre-Market (04:00–09:00 ET / HKT)  
> **Out of Scope:** Live intraday scanning, post-market scanning, execution of trades

---

## 1. EPIC Description

As a **trading system operator**, I want the system to automatically discover and catalog pre-market gap-up candidates across the entire tradeable universe, so that the AI analysis layer has a high-quality, filtered pool of opportunities to evaluate before market open.

### 1.1 Business Value

- **Opportunity Capture:** The best pre-market gaps form between 04:00–08:30 ET. Automated scanning ensures no candidate is missed due to human unavailability.
- **Noise Reduction:** Raw scanner output contains hundreds of symbols. The filter engine reduces this to a manageable, high-probability set.
- **Multi-Source Validation:** Using multiple independent data sources reduces false positives from any single source error.
- **HK Market Coverage:** Parallel scanning for Hong Kong equities ensures dual-market operational capability.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Scanning external data sources for pre-market price movements
  ✓ Enriching candidates with volume, fundamentals, and technical data
  ✓ Applying filter rules (blacklist, sector exclusion, OTC/ETF exclusion)
  ✓ Classifying catalysts and sentiment
  ✓ Persisting scan results with timestamps
  ✓ Supporting multiple scan passes per morning

OUT OF SCOPE:
  ✗ Executing trades based on scan results
  ✗ Intraday re-scanning after market open
  ✗ Options or futures scanning
  ✗ Fundamental analysis beyond catalyst classification
  ✗ AI decision-making (handled in EPIC-2)
```

## 2. Features

### Feature-1.1: Broad Universe Gap Scanner (FinViz-First)

#### Description
A high-performance scanner that queries a broad-market screener website to identify all US equities gapping up in pre-market. It serves as the primary discovery mechanism due to its comprehensive universe coverage.

#### User Stories

##### US-1.1.1: Early Pre-Market Discovery
> **As a** trading system,  
> **I want** to run a broad universe gap scan at 04:30 ET,  
> **So that** I capture overnight catalysts and early pre-market movers before they attract broader attention.

**Acceptance Criteria:**
- [ ] The scan completes within 15 minutes of initiation.
- [ ] The output contains at least 5 and at most 50 candidates.
- [ ] Each candidate includes: symbol, company name, exchange, sector, industry, previous close, pre-market price, gap percentage, gap dollar amount, pre-market volume, average daily volume, relative volume, float shares, float category, market cap, market cap category, ATR, RSI, SMA50, and data source.
- [ ] Candidates are sorted by gap percentage descending.
- [ ] The scan timestamp is recorded in America/New_York timezone.
- [ ] If the primary data source is unavailable, the scanner falls back to an alternative source within 30 seconds.

**Functional Validation:**
```
Input:  Scan time = 04:30 ET, min_gap = 2.0%, max_results = 50
Output: gapscan_finviz_YYYYMMDD_HHMMSS.json

Validation:
  1. File exists and is valid JSON.
  2. Contains "candidates" array with 5–50 items.
  3. Each candidate has all mandatory fields populated (non-null, non-empty).
  4. Gap percentages are between 2.0% and 1000%.
  5. Relative volume is a positive number.
  6. No OTC or ETF symbols are present (when exclusion is enabled).
  7. Scan duration is recorded and < 900 seconds.
```

##### US-1.1.2: Morning Refresh Scan
> **As a** trading system,  
> **I want** to run a second broad universe scan at 08:30 ET,  
> **So that** I capture late-breaking movers and detect trend direction (gap rising vs. gap fading) by comparing with the 04:30 scan.

**Acceptance Criteria:**
- [ ] The second scan uses identical parameters to the first scan.
- [ ] Candidates appearing in both scans include a trend indicator: RISING, STABLE, or FADING based on gap % change.
- [ ] New candidates in the second scan (not present in the first) are flagged as LATE_BREAKER.
- [ ] The combined scan data is persisted separately from the first scan.

**Functional Validation:**
```
Input:  Scan #1 at 04:30 ET, Scan #2 at 08:30 ET
Output: Two JSON files + trend comparison metadata

Validation:
  1. Both files exist and are valid.
  2. Symbols present in both scans have a "trend" field.
  3. Trend = RISING if gap % increased by > 0.5 percentage points.
  4. Trend = FADING if gap % decreased by > 0.5 percentage points.
  5. Trend = STABLE if gap % changed by ≤ 0.5 percentage points.
  6. Symbols unique to Scan #2 have "late_breaker": true.
```

### Feature-1.2: Enhanced Multi-Source Gap Scanner

#### Description
A deep enrichment scanner that captures comprehensive pre-market data including pre-market high/low (PMH/PML), VWAP, technical levels, short interest, and news sentiment. It aggregates data from multiple independent sources to maximize data completeness.

#### User Stories

##### US-1.2.1: Comprehensive Pre-Market Data Capture
> **As a** trading system,  
> **I want** to enrich each gap candidate with pre-market session data (PMH, PML, VWAP), technical levels, and short interest,  
> **So that** the AI analyzer has sufficient context to distinguish high-quality setups from low-quality ones.

**Acceptance Criteria:**
- [ ] Each enhanced candidate includes: pre-market high, pre-market low, pre-market open, pre-market VWAP, previous day high/low, 52-week high/low, distance to PMH/PML, distance to 52-week extremes.
- [ ] Volume metrics include: pre-market volume, 30-day average volume, 10-day average volume, relative volume.
- [ ] Fundamental metrics include: market cap, float shares, sector, industry, short ratio, short % of float.
- [ ] News metrics include: up to 5 recent headlines, sentiment score (-1.0 to 1.0), sentiment label (bullish/bearish/neutral), primary catalyst classification.
- [ ] A data completeness score (0–100%) is calculated based on the fraction of optional fields successfully populated.
- [ ] Candidates with data completeness < 50% are flagged as LOW_QUALITY but still included.

**Functional Validation:**
```
Input:  Symbol list from broad scanner
Output: Enhanced candidate list with enrichment fields

Validation:
  1. PMH ≥ pre-market price ≥ PML for all candidates.
  2. Distance to PMH is a percentage between 0% and 100%.
  3. Relative volume > 0.
  4. Sentiment score is between -1.0 and 1.0.
  5. Catalyst is one of: earnings, fda, merger, acquisition, partnership,
     upgrade, product, insider, offering, general.
  6. Data completeness is a percentage between 0% and 100%.
  7. At least 70% of candidates have data completeness ≥ 60%.
```

##### US-1.2.2: Multi-Source Data Validation
> **As a** trading system,  
> **I want** to cross-validate price and volume data across multiple sources,  
> **So that** single-source data errors do not contaminate the candidate pool.

**Acceptance Criteria:**
- [ ] When multiple sources report data for the same symbol, the scanner computes a consensus value.
- [ ] If source values differ by > 5%, the discrepancy is logged and the median value is used.
- [ ] Each candidate records which sources contributed data and the consensus methodology used.
- [ ] If all sources fail for a symbol, the symbol is excluded with a failure reason logged.

**Functional Validation:**
```
Input:  Same symbol from Source A and Source B
Output: Consensus candidate record

Validation:
  1. If |price_A - price_B| / price_A ≤ 5%, consensus = average.
  2. If |price_A - price_B| / price_A > 5%, consensus = median and
     discrepancy is logged.
  3. If only one source succeeds, consensus = that source and
     "single_source": true is flagged.
  4. If no sources succeed, symbol is excluded.
```

### Feature-1.3: HK Market Gap Scanner

#### Description
A dedicated gap scanner for Hong Kong equities that operates on HKT schedule and handles HK-specific market conventions (e.g., numeric tickers, .HK suffix, different pre-market hours).

#### User Stories

##### US-1.3.1: HK Pre-Market Discovery
> **As a** trading system operating in dual markets,  
> **I want** to scan HK equities for pre-market gaps at 09:00 HKT,  
> **So that** I have actionable HK candidates before the HK market opens at 09:30 HKT.

**Acceptance Criteria:**
- [ ] The HK scanner operates independently of the US scanner.
- [ ] The scanner handles HK ticker conventions (e.g., 0700.HK, 3690.HK).
- [ ] Output fields match the US enhanced scanner where applicable, with HK-specific adjustments.
- [ ] The scan respects HK market holidays (different from US holidays).
- [ ] Results are persisted with HKT timestamps and a "market": "HK" label.

**Functional Validation:**
```
Input:  HK symbol universe, scan time = 09:00 HKT
Output: gapscan_hk_YYYYMMDD_HHMMSS.json

Validation:
  1. All symbols end with ".HK" or are valid HK numeric codes.
  2. Timestamps are in HKT (UTC+8).
  3. No US symbols are mixed into HK results.
  4. HK market holidays are respected (skip scan on HK holidays).
```

### Feature-1.4: Filter Rules & Symbol Blacklist

#### Description
A centralized filter engine that applies business rules to exclude undesirable symbols from the candidate pool. Rules include symbol blacklists, sector exclusions, OTC/ETF exclusion, and sentiment-based filtering.

#### User Stories

##### US-1.4.1: Blacklist Enforcement
> **As a** risk manager,  
> **I want** to maintain a blacklist of symbols that must never be traded,  
> **So that** the system automatically excludes these symbols regardless of how attractive their gap appears.

**Acceptance Criteria:**
- [ ] Blacklisted symbols are excluded at the earliest stage of the pipeline.
- [ ] The blacklist supports exact symbol matching (case-insensitive).
- [ ] Changes to the blacklist take effect on the next scan without system restart.
- [ ] Every exclusion is logged with the reason "BLACKLIST".
- [ ] The blacklist includes a default set of known problematic symbols (e.g., leveraged ETFs, highly illiquid names).

**Functional Validation:**
```
Input:  Scan results containing blacklisted symbol "XYZ"
Output: Filtered scan results

Validation:
  1. "XYZ" does not appear in output candidates.
  2. Log entry exists: "Excluded XYZ: BLACKLIST".
  3. Adding "ABC" to blacklist and re-running scan excludes "ABC".
```

##### US-1.4.2: Sector Exclusion
> **As a** portfolio manager,  
> **I want** to exclude entire sectors from consideration,  
> **So that** I can avoid sectors with known structural risks or regulatory overhang.

**Acceptance Criteria:**
- [ ] Sector exclusions are configurable via a list of sector names.
- [ ] A candidate is excluded if its sector (case-insensitive) matches any excluded sector.
- [ ] Excluded sectors are logged with the reason "SECTOR_EXCLUSION".

**Functional Validation:**
```
Input:  Sector exclusion list = ["Biotechnology"], candidate sector = "Biotechnology"
Output: Candidate excluded

Validation:
  1. Candidate with sector "Biotechnology" is excluded.
  2. Candidate with sector "biotechnology" (lowercase) is excluded.
  3. Candidate with sector "Technology" is included.
```

##### US-1.4.3: OTC and ETF Exclusion
> **As a** risk manager,  
> **I want** to optionally exclude OTC (over-the-counter) stocks and ETFs,  
> **So that** I avoid illiquid or instrument-type mismatches.

**Acceptance Criteria:**
- [ ] When OTC exclusion is enabled, candidates with "is_otc": true are excluded.
- [ ] When ETF exclusion is enabled, candidates with "is_etf": true are excluded.
- [ ] Each exclusion is logged with reason "OTC_EXCLUDED" or "ETF_EXCLUDED".

**Functional Validation:**
```
Input:  Candidate A: is_otc=true; Candidate B: is_etf=true; OTC/ETF exclusion enabled
Output: Both A and B excluded

Validation:
  1. Candidate A excluded with reason "OTC_EXCLUDED".
  2. Candidate B excluded with reason "ETF_EXCLUDED".
  3. When exclusion is disabled, both candidates are included.
```

##### US-1.4.4: Catalyst Quality Scoring
> **As a** trading system,  
> **I want** to classify and score the catalyst driving each gap,  
> **So that** low-quality or ambiguous catalysts can be deprioritized.

**Acceptance Criteria:**
- [ ] Each candidate is assigned a primary catalyst from a predefined taxonomy.
- [ ] Each catalyst category has a quality score (1–10) reflecting historical predictability.
- [ ] Catalyst categories include: earnings (10), FDA (10), merger (9), acquisition (9), partnership (6), upgrade (7), product (5), insider (3), offering (1), general (2).
- [ ] The catalyst score contributes to the overall AI scoring in EPIC-2.

**Functional Validation:**
```
Input:  News headline: "ABC beats Q1 earnings estimates"
Output: catalyst="earnings", score=10

Validation:
  1. Keyword "earnings" matches earnings category.
  2. Keyword "beats" reinforces earnings category.
  3. Score = 10 (highest quality).
```

##### US-1.4.5: Sentiment Analysis
> **As a** trading system,  
> **I want** to compute a sentiment score for each candidate based on recent news headlines,  
> **So that** candidates with overwhelmingly negative sentiment can be flagged or excluded.

**Acceptance Criteria:**
- [ ] Sentiment is computed using weighted keyword matching against positive and negative word lists.
- [ ] The sentiment score ranges from -1.0 (maximally bearish) to +1.0 (maximally bullish).
- [ ] The sentiment label is: "bullish" if score > 0.2, "bearish" if score < -0.2, "neutral" otherwise.
- [ ] Headlines with no sentiment keywords return "neutral" with score 0.0.
- [ ] The sentiment analysis is deterministic: identical headlines always produce identical scores.

**Functional Validation:**
```
Input:  Headlines: ["ABC beats earnings, stock surges", "XYZ misses revenue, plunges"]
Output: ABC: score=+0.67, label="bullish"; XYZ: score=-0.67, label="bearish"

Validation:
  1. Positive words (beats, surges) weighted correctly.
  2. Negative words (misses, plunges) weighted correctly.
  3. Neutral headline returns score=0.0, label="neutral".
```

## 3. Functional Requirements

### FR-1.1: Universe Coverage
The broad universe scanner must cover all NASDAQ, NYSE, and AMEX equities that meet minimum price and volume thresholds. The minimum tradeable price is $0.50; the maximum is $1,000.

### FR-1.2: Scan Frequency
At least two scan passes must run each trading morning: one early (04:30 ET) and one refresh (08:30 ET). Additional ad-hoc scans may be triggered by the operator.

### FR-1.3: Result Persistence
Every scan result must be persisted with: scan timestamp, data source(s), total scanned count, total passed count, candidate list, scan duration, and any errors encountered.

### FR-1.4: Error Handling
If a data source returns partial data, the scanner must populate available fields and flag missing fields. If a data source is entirely unavailable, the scanner must log the failure and attempt fallback sources.

### FR-1.5: Configurable Thresholds
The following thresholds must be externally configurable:
- Minimum gap percentage (default: 3.0%)
- Maximum gap percentage (default: 1000.0%)
- Minimum pre-market volume (default: 100,000 shares)
- Minimum relative volume (default: 0.0)
- Maximum candidates per scan (default: 50)
- Exchange filter (default: NASDAQ, NYSE, AMEX)

### FR-1.6: Data Freshness
Scan data must be timestamped and considered valid for 24 hours. Scans older than 24 hours must trigger a warning if used by downstream components.

## 4. Acceptance Criteria (EPIC Level)

- [ ] **AC-1.1:** On a typical trading morning, the system produces at least one complete scan result with ≥ 5 candidates before 09:00 ET.
- [ ] **AC-1.2:** No blacklisted symbol, OTC symbol (when excluded), or ETF (when excluded) appears in any final scan output.
- [ ] **AC-1.3:** Every candidate in the enhanced scanner has a data completeness score ≥ 50%.
- [ ] **AC-1.4:** The HK scanner produces valid results on HK trading days without interfering with the US scanner schedule.
- [ ] **AC-1.5:** Scan results are persisted within 60 seconds of scan completion and are retrievable by date and time range.
