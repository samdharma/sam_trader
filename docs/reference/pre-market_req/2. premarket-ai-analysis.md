# EPIC-2: AI-Powered Pre-Market Analysis & Recommendation

> **EPIC ID:** EPIC-2  
> **Status:** Active  
> **Phase:** Pre-Market (09:00 ET, after gap scans complete)  
> **Depends On:** EPIC-1 (Gap Discovery & Scanning)  
> **Out of Scope:** Live intraday AI decisions (post-market-open), trade execution, backtest engine

---

## 1. EPIC Description

As a **trading system operator**, I want the system to intelligently analyze every pre-market gap candidate, score its tradeability, and construct an optimal daily portfolio allocation, so that market operations receive validated, ranked, and sized recommendations rather than raw scanner output.

### 1.1 Business Value

- **Decision Quality:** AI analysis transforms raw gap data into structured trade recommendations with explicit reasoning, confidence, and risk parameters.
- **Portfolio Context:** The AI Portfolio Manager ensures recommendations respect existing positions, sector concentration limits, and cash reserve targets.
- **Explainability:** Every recommendation includes a natural-language reasoning narrative and a structured score breakdown, enabling operator review and audit.
- **Speed:** The entire analysis pipeline must complete within 10 minutes for up to 50 candidates.
- **Graceful Degradation:** If AI services are unavailable, rule-based fallback analysis still produces actionable recommendations.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Per-candidate AI analysis and scoring
  ✓ Portfolio-level capital allocation and rebalancing
  ✓ ReAct agentic reasoning for complex decisions
  ✓ Rule-based fast-path advisor for deterministic scenarios
  ✓ Structured output with confidence, sizing, stops, and targets
  ✓ Fallback to rule-based analysis when AI is unavailable

OUT OF SCOPE:
  ✗ Actual order submission or trade execution
  ✗ Intraday position management (handled by intraday monitor)
  ✗ Historical backtesting (handled by backtest engine)
  ✗ Market data ingestion (handled in EPIC-1)
  ✗ Position sizing arithmetic (handled in EPIC-3)
```

## 2. Features

### Feature-2.1: AI Gap Candidate Analyzer

#### Description
An intelligent analysis module that consumes enriched gap candidates and produces per-symbol trading recommendations. It evaluates gap quality, technical setup, sentiment, liquidity, risk, and market context to generate a composite score and a recommendation grade.

#### User Stories

##### US-2.1.1: Per-Candidate AI Scoring
> **As a** trading system,  
> **I want** to analyze each gap candidate across six dimensions and produce a total score from 0 to 100,  
> **So that** candidates can be objectively ranked and filtered.

**Acceptance Criteria:**
- [ ] Each candidate receives scores across six dimensions:
  - Gap Quality (0–25): Magnitude, consistency across scans, distance to 52-week high
  - Technical Setup (0–20): PMH/PML relationship, ATR context, RSI level, support/resistance alignment
  - Sentiment (0–20): News sentiment score, headline count, source credibility
  - Liquidity (0–15): Relative volume, float category, dollar volume, spread estimate
  - Risk (0–10): Short interest, volatility regime alignment, sector risk
  - Market Context (0–10): Sector leadership, peer correlation, index alignment
- [ ] Total score is the sum of all six dimensions (0–100).
- [ ] Candidates with total score < 40 are automatically graded AVOID.
- [ ] Candidates with total score ≥ 80 are candidates for STRONG_BUY.
- [ ] The score breakdown is included in the output for auditability.

**Functional Validation:**
```
Input:  Enhanced gap candidate for symbol "AAPL"
Output: AIRecommendation record

Validation:
  1. All six dimension scores are present and within their ranges.
  2. Total score = sum of dimension scores.
  3. If total_score < 40, recommendation = "AVOID".
  4. If total_score ≥ 80 and confidence ≥ 0.7, recommendation = "STRONG_BUY".
  5. Output includes natural-language reasoning (≥ 50 characters).
  6. Output includes key_factors list (≥ 1 item) and risk_factors list (≥ 0 items).
```

##### US-2.1.2: Recommendation Grading
> **As a** trading system,  
> **I want** to classify each candidate into one of five recommendation grades,  
> **So that** downstream components know how aggressively to act.

**Acceptance Criteria:**
- [ ] Recommendation grades are:
  - **STRONG_BUY:** Highest confidence, strong catalyst, favorable technicals, good liquidity. Requires total score ≥ 80 and confidence ≥ 0.7.
  - **BUY:** Solid setup with manageable risk. Requires total score ≥ 60 and confidence ≥ 0.5.
  - **CAUTIOUS:** Mixed signals or elevated risk. Score 40–59 or confidence 0.3–0.5.
  - **WATCH:** Interesting but not actionable today. Score < 40 with notable catalyst.
  - **AVOID:** Poor setup, bad risk/reward, or red flags. Score < 40.
- [ ] The grade is derived deterministically from score and confidence thresholds.
- [ ] A "conviction" field (STRONG, MODERATE, WEAK) is included for additional granularity.

**Functional Validation:**
```
Input:  Candidate with total_score=85, confidence=0.75, gap_quality=22
Output: recommendation="STRONG_BUY", conviction="STRONG"

Validation:
  1. STRONG_BUY requires total_score ≥ 80 AND confidence ≥ 0.7.
  2. BUY requires total_score ≥ 60 AND confidence ≥ 0.5.
  3. CAUTIOUS requires total_score ≥ 40 OR confidence ≥ 0.3.
  4. AVOID is default for all other cases.
  5. Conviction mapping: confidence ≥ 0.7 → STRONG; 0.4–0.69 → MODERATE; < 0.4 → WEAK.
```

##### US-2.1.3: Suggested Trade Parameters
> **As a** trading system,  
> **I want** the AI analyzer to suggest entry price, stop-loss price, take-profit price, and position size percentage,  
> **So that** the position sizer (EPIC-3) has intelligent starting parameters.

**Acceptance Criteria:**
- [ ] Suggested entry price is within 1% of the current pre-market price.
- [ ] Suggested stop price is based on ATR or pre-market low, whichever is more conservative.
- [ ] Suggested target price implies a risk-reward ratio of at least 1.5:1.
- [ ] Suggested position size is expressed as a percentage of portfolio equity (0%–15%).
- [ ] For STRONG_BUY recommendations, suggested size is 5%–15%.
- [ ] For BUY recommendations, suggested size is 2%–8%.
- [ ] For CAUTIOUS recommendations, suggested size is 0%–3%.
- [ ] AVOID and WATCH recommendations have suggested size = 0%.

**Functional Validation:**
```
Input:  Candidate with ATR=$2.50, premarket_price=$150.00, PMH=$152.00, PML=$148.00
Output: suggested_entry=$150.00, suggested_stop=$147.50 (1 ATR below entry),
        suggested_target=$154.00 (RR = 1.6:1)

Validation:
  1. suggested_entry within 1% of premarket_price.
  2. suggested_stop ≤ min(premarket_price - 1.5*ATR, PML).
  3. (suggested_target - suggested_entry) / (suggested_entry - suggested_stop) ≥ 1.5.
  4. Position size % is within grade-appropriate range.
```

##### US-2.1.4: AI Fallback to Rule-Based Analysis
> **As a** trading system operator,  
> **I want** the system to fall back to rule-based analysis when AI services are unavailable,  
> **So that** the pre-market pipeline never produces zero recommendations due to external service outages.

**Acceptance Criteria:**
- [ ] If the primary AI provider is unavailable, the system attempts secondary providers in a defined priority order.
- [ ] If no AI provider is available, rule-based analysis is applied.
- [ ] Rule-based analysis uses the same scoring dimensions but with deterministic formulas instead of LLM inference.
- [ ] Rule-based recommendations are flagged with "llm_used": "RuleBased" for transparency.
- [ ] The pipeline still produces a valid execution bundle even in full fallback mode.

**Functional Validation:**
```
Input:  All AI providers unreachable
Output: Rule-based recommendations

Validation:
  1. Recommendations are generated within 5 seconds per candidate.
  2. Each recommendation has all required fields populated.
  3. "llm_used" field = "RuleBased".
  4. Pipeline completes successfully and produces execution bundle.
```

### Feature-2.2: AI Portfolio Manager & Capital Allocator

#### Description
A portfolio-level AI module that constructs the optimal daily target portfolio by considering AI recommendations, current open positions, risk constraints, sector limits, and cash reserve targets. It outputs rebalancing orders to align the actual portfolio with the target.

#### User Stories

##### US-2.2.1: Daily Portfolio Construction
> **As a** trading system,  
> **I want** to construct a target portfolio every morning before market open,  
> **So that** capital is allocated optimally across the best opportunities while respecting risk constraints.

**Acceptance Criteria:**
- [ ] The portfolio manager reads current positions and their market values.
- [ ] It reads all AI recommendations with scores ≥ 60.
- [ ] It applies risk constraints:
  - Maximum 10 open positions
  - Maximum 15% of NAV in any single position
  - Maximum 25% of NAV in any single sector
  - Minimum 15% cash reserve
- [ ] It constructs a target portfolio: symbol → target dollar amount.
- [ ] It compares target to current holdings and generates rebalancing orders:
  - BUY: New position or increasing existing position
  - SELL: Reducing or closing existing position
  - TRIM: Reducing overweight position
- [ ] The output includes reasoning for each allocation decision.

**Functional Validation:**
```
Input:  Account NAV = $100,000; Current positions: AAPL=$10K, TSLA=$5K
        AI recommendations: MSFT (score 85), NVDA (score 78), AMD (score 72)
Output: Portfolio recommendation

Validation:
  1. Cash target ≥ $15,000 (15% of NAV).
  2. No single target position > $15,000.
  3. No sector allocation > $25,000.
  4. Total target positions + cash = $100,000 (± $100 rounding).
  5. Rebalancing orders exist for deviations > $1,000.
```

##### US-2.2.2: Sector Concentration Guard
> **As a** risk manager,  
> **I want** the portfolio manager to enforce sector concentration limits,  
> **So that** a single sector downturn cannot disproportionately damage the portfolio.

**Acceptance Criteria:**
- [ ] Each candidate is assigned a sector based on its sector field.
- [ ] The sum of target dollar amounts within any sector cannot exceed 25% of NAV.
- [ ] If adding a new candidate would violate sector limits, it is either:
  - Sized down to fit within the limit, or
  - Excluded if it cannot be sized meaningfully (> $500)
- [ ] Sector limit breaches are logged with the reason "SECTOR_LIMIT".

**Functional Validation:**
```
Input:  NAV = $100K; Sector limit = 25%; Technology sector already = $23K;
        New candidate NVDA (Technology) with target $5K
Output: NVDA sized down to $2K or excluded

Validation:
  1. Final Technology allocation ≤ $25K.
  2. If NVDA is sized down, new target is $2K.
  3. If NVDA is excluded, reason = "SECTOR_LIMIT".
```

##### US-2.2.3: Cash Reserve Management
> **As a** portfolio manager,  
> **I want** the system to hold a configurable cash reserve,  
> **So that** there is dry powder for new opportunities during the trading day and a buffer against drawdowns.

**Acceptance Criteria:**
- [ ] The cash reserve target is configurable (default: 15% of NAV).
- [ ] The portfolio manager never allocates more than (NAV - cash_reserve) to positions.
- [ ] If rebalancing would reduce cash below the target, BUY orders are reduced or skipped.
- [ ] The cash reserve percentage is included in the portfolio recommendation output.

**Functional Validation:**
```
Input:  NAV = $100K; Cash reserve target = 20%; Current cash = $10K
        AI recommendations suggest $95K in new positions
Output: Positions sized so total allocation = $80K, cash = $20K

Validation:
  1. Sum of all target position values ≤ $80,000.
  2. Cash target = $20,000 (20% of NAV).
  3. If only $70K of recommendations are actionable, remaining $10K stays in cash.
```

### Feature-2.3: ReAct Agentic Reasoning Framework

#### Description
A reasoning-and-acting framework that allows the AI to iteratively gather information, reason about it, and take actions (e.g., researching a symbol, checking risk limits, fetching backtest history) before producing a final recommendation. This enables complex, context-aware decisions.

#### User Stories

##### US-2.3.1: Tool-Augmented Analysis
> **As a** trading system,  
> **I want** the AI to use specialized tools to gather additional context before making recommendations,  
> **So that** recommendations are grounded in real data rather than static patterns.

**Acceptance Criteria:**
- [ ] The ReAct agent has access to tools including:
  - Symbol research (earnings context, peer comparison, insider activity)
  - Backtest history (recent performance for the symbol)
  - Risk limit check (current circuit breaker status, portfolio heat)
  - Market regime lookup (current HMM regime)
  - Current price lookup (latest available price)
- [ ] The agent performs up to 3 reasoning iterations per symbol.
- [ ] Each iteration produces a thought, action, and observation.
- [ ] The final recommendation references specific observations from the tool calls.
- [ ] All tool calls are logged with inputs, outputs, and timing.

**Functional Validation:**
```
Input:  Symbol "AAPL" with ambiguous technical setup
Output: ReAct trace with 2–3 steps, final recommendation

Validation:
  1. Trace contains at least one tool call.
  2. Final recommendation references specific data points from tool outputs.
  3. Total reasoning time < 5 seconds per symbol.
  4. All steps are persisted for audit.
```

##### US-2.3.2: Token Budget Management
> **As a** system operator,  
> **I want** the AI analysis to respect a token budget,  
> **So that** daily AI costs remain predictable and bounded.

**Acceptance Criteria:**
- [ ] A token budget tracker estimates token usage before each LLM call.
- [ ] If a call would exceed the daily budget, the agent falls back to rule-based analysis.
- [ ] Daily token usage is logged and reported in the pipeline summary.
- [ ] The daily budget is configurable (default: equivalent to ~$5 USD in API costs).

**Functional Validation:**
```
Input:  Daily budget = 100,000 tokens; Current usage = 95,000 tokens
        New request estimates 10,000 tokens
Output: Fallback to rule-based analysis

Validation:
  1. No LLM call is made when budget would be exceeded.
  2. Rule-based recommendation is produced instead.
  3. Log entry: "Budget exceeded, fallback to RuleBased".
```

### Feature-2.4: Rule-Based Fast Path Advisor

#### Description
A deterministic, arithmetic-only advisor optimized for sub-millisecond decisions. It applies fixed rules to open positions to generate immediate action signals without LLM latency. Used as a fallback and for high-frequency intraday decisions.

#### User Stories

##### US-2.4.1: Fast Position Evaluation
> **As a** trading system,  
> **I want** to evaluate open positions using deterministic arithmetic rules,  
> **So that** I can generate adjustment signals without waiting for LLM inference.

**Acceptance Criteria:**
- [ ] The fast advisor evaluates positions using these rules in priority order:
  1. If unrealized P&L ≤ -1.5 × ATR → signal FULL_EXIT
  2. If current price ≥ target price × 0.95 → signal PARTIAL_EXIT
  3. If unrealized P&L ≥ 2.0 × ATR → signal TIGHTEN_STOP
  4. If regime is BEARISH and unrealized P&L > 0 → signal TIGHTEN_STOP
  5. Otherwise → signal HOLD
- [ ] Each signal includes: symbol, action, confidence (0–100), reasoning, current price.
- [ ] Evaluation time per position is < 1 millisecond.
- [ ] All rule thresholds are configurable.

**Functional Validation:**
```
Input:  Position: AAPL, qty=100, avg_entry=$150, current_price=$145,
        unrealized_pl=-$500, ATR=$3.00, target_price=$160, regime="LOW_VOL"
Output: AdjustmentSignal(action="FULL_EXIT", confidence=95,
        reasoning="Unrealized P&L $-500 ≤ -1.5×ATR ($-4.50)...")

Validation:
  1. Rule 1 triggers because -500 ≤ -1.5 * 3.00 (wait, this example is wrong,
     should be -4.50 vs -500 — let me fix: ATR should be $333 or unrealized_pl
     should be -$5. Let's use: unrealized_pl=-$500, ATR=$333 is wrong.
     Better: unrealized_pl=-$500, ATR=$300... no.
     Let's say: ATR=$2.50, unrealized_pl=-$500... that would be 200×ATR.
     Let's just validate the logic:)
  
  Corrected validation:
  Input:  AAPL, unrealized_pl=-$5.00, ATR=$3.00
  Output: FULL_EXIT (because -5.00 ≤ -4.50)
  
  Input:  AAPL, current_price=$155, target=$160
  Output: PARTIAL_EXIT (because 155 ≥ 160 * 0.95 = 152)
  
  Input:  AAPL, unrealized_pl=$7.00, ATR=$3.00
  Output: TIGHTEN_STOP (because 7.00 ≥ 6.00)
  
  Input:  AAPL, regime="BEARISH", unrealized_pl=$1.00
  Output: TIGHTEN_STOP
  
  Input:  None of above conditions met
  Output: HOLD
```

## 3. Functional Requirements

### FR-2.1: Analysis Completeness
Every candidate that passes filtering (EPIC-1) must receive an AI or rule-based analysis. No candidate may be silently dropped.

### FR-2.2: Deterministic Grading
The mapping from total score + confidence to recommendation grade must be deterministic and documented. Randomness (if any) must be bounded and logged.

### FR-2.3: Portfolio Constraint Enforcement
The portfolio manager must never produce a recommendation that violates hard constraints (max positions, max single position %, max sector %, min cash %). If constraints make a recommendation impossible, the portfolio manager must log the conflict and produce the best feasible alternative.

### FR-2.4: Explainability
Every AI recommendation must include:
- A natural-language reasoning string (minimum 50 characters)
- A list of key decision factors (minimum 1 item)
- A list of identified risk factors (may be empty)
- The LLM provider used (or "RuleBased")
- A timestamp and trace identifier

### FR-2.5: Fallback Chain
The system must attempt AI providers in priority order. If all fail, it must fall back to rule-based analysis within 5 seconds. The fallback must never produce an empty recommendation set unless zero candidates were input.

### FR-2.6: Traceability
Every AI recommendation and portfolio decision must carry the same trace identifier as the pre-market pipeline that produced it, enabling end-to-end audit.

## 4. Acceptance Criteria (EPIC Level)

- [ ] **AC-2.1:** For a typical morning with 20–50 candidates, the AI analyzer produces at least 3 actionable recommendations (BUY or STRONG_BUY) with scores ≥ 60.
- [ ] **AC-2.2:** Every recommendation includes a complete score breakdown, reasoning narrative, and suggested trade parameters.
- [ ] **AC-2.3:** The portfolio manager produces a target portfolio that satisfies all hard constraints (max positions, max position %, max sector %, min cash %).
- [ ] **AC-2.4:** When all AI providers are unavailable, the system produces rule-based recommendations for all candidates within 30 seconds.
- [ ] **AC-2.5:** The ReAct agent completes analysis for a single candidate within 5 seconds, using no more than 3 tool calls.
- [ ] **AC-2.6:** Token budget is enforced; exceeding the budget triggers rule-based fallback without system error.
