# EPIC-4: Market Regime Detection & Adaptation

> **EPIC ID:** EPIC-4  
> **Status:** Active  
> **Phase:** Pre-Market (runs during pipeline, typically 09:00 ET)  
> **Markets:** US (primary), HK (secondary with separate model)  
> **Out of Scope:** Intraday regime updates (handled by intraday monitor), backtest regime simulation, futures/options regime detection

---

## 1. EPIC Description

As a **trading system operator**, I want the system to classify the current market state into a volatility regime before market open, so that position sizes, stop distances, and strategy selection can be adapted to the prevailing market conditions.

### 1.1 Business Value

- **Adaptive Sizing:** In low-volatility regimes, the system can take larger positions with wider stops. In high-volatility regimes, positions are smaller and stops are tighter.
- **Strategy Selection:** Different strategies perform better in different regimes. Regime awareness enables strategy matching.
- **Risk Calibration:** Portfolio heat limits and circuit breaker thresholds can be adjusted based on regime.
- **Predictive Edge:** Regime persistence (tendency to stay in the same regime for multiple days) provides a short-term forecast of market behavior.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Daily regime classification before market open
  ✓ Regime persistence and stability detection
  ✓ Dual-venue support (US model, HK model)
  ✓ Regime-aware sizing multipliers
  ✓ Regime-aware strategy adaptation
  ✓ Model versioning and persistence
  ✓ Historical regime tracking

OUT OF SCOPE:
  ✗ Real-time intraday regime updates (every tick)
  ✗ Regime prediction beyond the current day
  ✗ Multi-asset regime detection (crypto, forex, commodities)
  ✗ News-based regime classification (sentiment is handled in EPIC-2)
```

## 2. Features

### Feature-4.1: HMM-Based Regime Detection

#### Description
A statistical regime detection engine that uses Hidden Markov Models to classify market state based on historical price features (returns, volatility, volume). It uses only the forward algorithm (no look-ahead) to prevent future-information bias.

#### User Stories

##### US-4.1.1: Daily Regime Classification
> **As a** trading system,  
> **I want** to classify today's market regime before the opening bell,  
> **So that** all pre-market decisions (sizing, strategy, risk) are calibrated to the current environment.

**Acceptance Criteria:**
- [ ] The regime engine classifies the market into one of four regimes:
  - **LOW_VOL:** Low volatility, trending or range-bound. Favorable for larger positions.
  - **MID_VOL:** Moderate volatility, typical market conditions. Normal position sizes.
  - **HIGH_VOL:** Elevated volatility, choppy conditions. Reduced position sizes, tighter stops.
  - **BEARISH:** Downward trending with high volatility. Defensive posture, minimal new entries.
- [ ] Classification uses the forward algorithm only (no Viterbi) to ensure no look-ahead bias.
- [ ] The engine requires a minimum of 252 training bars (1 year of daily data).
- [ ] Classification includes a confidence score (0.0–1.0) representing the probability of the most likely state.
- [ ] If confidence < 0.6, the regime is reported as UNKNOWN with a warning.
- [ ] The classification completes within 30 seconds.

**Functional Validation:**
```
Input:  1 year of daily SPY data (returns, volatility, volume)
Output: RegimePrediction

Validation:
  1. Regime is one of: LOW_VOL, MID_VOL, HIGH_VOL, BEARISH, UNKNOWN.
  2. Confidence is between 0.0 and 1.0.
  3. If confidence < 0.6, regime = UNKNOWN.
  4. Classification time < 30 seconds.
  5. State probabilities sum to 1.0 (± 0.001).
```

##### US-4.1.2: Model Selection and Training
> **As a** trading system,  
> **I want** to automatically select the optimal number of hidden states for the HMM,  
> **So that** the model complexity matches the data without overfitting.

**Acceptance Criteria:**
- [ ] The engine tests candidate state counts from 3 to 7.
- [ ] For each candidate, it trains multiple random initializations (default: 10) and selects the best fit.
- [ ] Model selection uses the Bayesian Information Criterion (BIC) to balance fit and complexity.
- [ ] The best model is persisted with a version identifier and training date.
- [ ] If no model exists, the engine auto-trains on the available historical data.
- [ ] If training data is insufficient (< 252 bars), the engine reports UNKNOWN and schedules retraining.

**Functional Validation:**
```
Input:  2 years of daily feature data
Output: Trained HMM model with optimal state count

Validation:
  1. Candidate state counts tested: 3, 4, 5, 6, 7.
  2. BIC values are computed for each candidate.
  3. Selected model has the lowest BIC.
  4. Model file is saved with timestamp and version.
  5. Model load time < 5 seconds.
```

##### US-4.1.3: Regime Stability Detection
> **As a** trading system,  
> **I want** to know whether the current regime has stabilized or is flickering,  
> **So that** I can trust regime-based decisions only when the regime is stable.

**Acceptance Criteria:**
- [ ] A regime is considered **stable** if it has persisted for at least a configurable number of consecutive bars (default: 3 days).
- [ ] Regime flickering (rapid switching between regimes) triggers a warning.
- [ ] The stability flag is included in the regime prediction output.
- [ ] Unstable regimes are treated as MID_VOL for sizing purposes as a conservative default.

**Functional Validation:**
```
Input:  Regime history: [LOW_VOL, LOW_VOL, LOW_VOL] (last 3 days)
Output: Regime = LOW_VOL, is_stable = true

Validation:
  1. LOW_VOL persisted for 3 days ≥ stability threshold (3).
  2. is_stable = true.

Input:  Regime history: [LOW_VOL, HIGH_VOL, LOW_VOL] (last 3 days)
Output: Regime = LOW_VOL, is_stable = false

Validation:
  1. Regime switched within last 3 days.
  2. is_stable = false.
  3. Sizing uses MID_VOL multiplier as conservative fallback.
```

##### US-4.1.4: Dual-Venue Regime Support
> **As a** dual-market trading system,  
> **I want** separate regime models for US and HK markets,  
> **So that** regime classification is accurate for each market's unique characteristics.

**Acceptance Criteria:**
- [ ] The US regime model is trained on US market index data (e.g., SPY).
- [ ] The HK regime model is trained on HK market index data (e.g., HSI).
- [ ] Each model is persisted separately with venue-specific naming.
- [ ] The engine selects the correct model based on the market venue parameter.
- [ ] US and HK regimes are computed independently and do not influence each other.

**Functional Validation:**
```
Input:  venue="US", features from SPY
Output: US regime prediction

Validation:
  1. US model is loaded.
  2. Prediction is based on US features only.

Input:  venue="HK", features from HSI
Output: HK regime prediction

Validation:
  1. HK model is loaded.
  2. Prediction is based on HK features only.
  3. US and HK predictions may differ.
```

### Feature-4.2: Regime-Aware Strategy Adaptation

#### Description
A decision layer that adjusts trading parameters based on the detected regime. It modifies position sizing multipliers, stop-loss distances, strategy selection, and risk thresholds.

#### User Stories

##### US-4.2.1: Regime-Based Sizing Multipliers
> **As a** trading system,  
> **I want** to adjust position sizes based on the current volatility regime,  
> **So that** I take smaller positions in volatile markets and larger positions in calm markets.

**Acceptance Criteria:**
- [ ] Regime sizing multipliers:
  - LOW_VOL: 1.25× base size (opportunity to capitalize on predictable moves)
  - MID_VOL: 1.0× base size (normal operation)
  - HIGH_VOL: 0.60× base size (defensive reduction)
  - BEARISH: 0.50× base size (minimal exposure)
  - UNKNOWN: 1.0× base size (conservative default)
- [ ] The multiplier is applied after Kelly calculation but before the hard risk ceiling.
- [ ] The applied multiplier is logged in the position size reasoning.

**Functional Validation:**
```
Input:  Base size = $5,000; Regime = HIGH_VOL
Output: Adjusted size = $5,000 × 0.60 = $3,000

Validation:
  1. Multiplier correctly applied.
  2. Adjusted size ≤ base size for HIGH_VOL.
  3. Reasoning includes "HIGH_VOL regime: size reduced by 40%".

Input:  Base size = $5,000; Regime = LOW_VOL
Output: Adjusted size = $5,000 × 1.25 = $6,250

Validation:
  1. Multiplier correctly applied.
  2. Adjusted size ≥ base size for LOW_VOL.
  3. Adjusted size still ≤ max position limit.
```

##### US-4.2.2: Regime-Based Stop Adjustment
> **As a** risk manager,  
> **I want** to tighten or loosen stop distances based on regime,  
> **So that** stops are appropriate for the current volatility level.

**Acceptance Criteria:**
- [ ] Stop distance adjustments by regime:
  - LOW_VOL: 2.0× ATR (wider stops to avoid noise)
  - MID_VOL: 1.5× ATR (standard)
  - HIGH_VOL: 1.0× ATR (tighter stops to limit damage)
  - BEARISH: 1.0× ATR or pre-market low, whichever is tighter
- [ ] The adjusted stop is used in position sizing and in the execution bundle.
- [ ] If the adjusted stop would exceed the pre-market low, the tighter value is used.

**Functional Validation:**
```
Input:  ATR = $2.00, premarket_price = $100, PML = $98, regime = HIGH_VOL
Output: stop = $100 - (1.0 × $2.00) = $98.00

Validation:
  1. HIGH_VOL uses 1.0× ATR.
  2. Stop = $98.00.
  3. If PML were $97.00, stop would still be $98.00 (ATR-based is tighter).

Input:  Same, regime = LOW_VOL
Output: stop = $100 - (2.0 × $2.00) = $96.00

Validation:
  1. LOW_VOL uses 2.0× ATR.
  2. Stop = $96.00.
  3. Stop is wider than HIGH_VOL case.
```

##### US-4.2.3: Regime-Aware Strategy Selection
> **As a** trading system,  
> **I want** to select or weight trading strategies based on regime,  
> **So that** the system uses strategies historically suited to current conditions.

**Acceptance Criteria:**
- [ ] Strategy weights by regime:
  - LOW_VOL: Favor trend-following and momentum strategies.
  - MID_VOL: Balanced mix of strategies.
  - HIGH_VOL: Favor mean-reversion and defensive strategies.
  - BEARISH: Favor cash preservation and short-biased strategies; minimize long entries.
- [ ] The strategy recommendation is included in the execution bundle.
- [ ] Strategy selection is advisory; the execution layer may override based on per-symbol analysis.

**Functional Validation:**
```
Input:  Regime = BEARISH, candidate = long-biased gap momentum setup
Output: Strategy recommendation = "defensive" or "gap_retest"
        Confidence reduction = 20%

Validation:
  1. BEARISH regime triggers defensive strategy preference.
  2. Long-biased setup confidence is reduced.
  3. Recommendation is logged in execution bundle.
```

## 3. Functional Requirements

### FR-4.1: No Look-Ahead Bias
The regime engine must use only the forward algorithm for inference. Viterbi or any method requiring future information is strictly prohibited.

### FR-4.2: Minimum Training Data
The engine must refuse to classify if fewer than 252 bars of historical data are available. It must report UNKNOWN and log the data deficiency.

### FR-4.3: Model Versioning
Every trained model must be persisted with: version number, training date, venue (US/HK), number of states, BIC score, and feature names used.

### FR-4.4: Regime Persistence
Regime predictions must be persisted daily with: date, regime, confidence, stability flag, state probabilities, and model version used.

### FR-4.5: Graceful Degradation
If the regime model is missing or corrupted, the system must default to MID_VOL with a warning. Trading must not be blocked solely due to regime detection failure.

### FR-4.6: Independent Venue Models
US and HK regime models must be completely independent. Training data, model parameters, and predictions must not cross-contaminate between venues.

## 4. Acceptance Criteria (EPIC Level)

- [ ] **AC-4.1:** The regime engine produces a valid regime classification within 30 seconds on every trading day.
- [ ] **AC-4.2:** The classification confidence is ≥ 0.6 for at least 80% of trading days.
- [ ] **AC-4.3:** Regime labels are stable (no flickering) for at least 70% of consecutive trading days.
- [ ] **AC-4.4:** LOW_VOL regime correctly applies a 1.25× sizing multiplier; HIGH_VOL correctly applies 0.60×.
- [ ] **AC-4.5:** When the model is missing, the system defaults to MID_VOL and logs a warning without blocking trading.
- [ ] **AC-4.6:** US and HK regime predictions are computed independently and may differ on the same calendar day.
