# CSAM Pre-Market Operations — Functional Overview

> **Scope:** This document covers all functional capabilities that execute *before* the US market opens at 09:30 ET (and HK market at 09:30 HKT). Everything after market open — live execution, intraday monitoring, order fills, and end-of-day close — is explicitly out of scope.
>
> **Audience:** Product owners, business analysts, QA engineers, and operators who need to understand what the pre-market phase does, why it exists, and how to validate it.
>
> **Version:** v3.7 · Date: 2026-05-17

---

## 1. Purpose & Business Context

The pre-market phase exists to answer one question before the opening bell:

> *"Given everything we know right now, which positions should we take today, how large should they be, and under what conditions should we enter?"*

Pre-market operations transform raw market data into an **actionable execution plan** that can be handed off to market operations with confidence. The system must do this autonomously, consistently, and with full auditability — every day, before every market open.

### 1.1 Key Business Drivers

| Driver | Why It Matters |
|--------|---------------|
| **Speed** | Pre-market gaps form between 04:00–09:30 ET. Late discovery means missed opportunity or worse entry prices. |
| **Quality** | Not all gaps are tradable. The system must distinguish genuine catalyst-driven gaps from noise, dilution, or manipulation. |
| **Risk Control** | Position sizing and portfolio heat must be computed *before* orders are submitted, not after. |
| **Autonomy** | The system runs on a fixed cron schedule without human intervention, but every decision must be explainable and reversible. |
| **Handoff Integrity** | The output of pre-market must be a single, validated artifact that market operations can consume without ambiguity. |

### 1.2 Daily Pre-Market Timeline (ET)

```
04:30  → Gap Scan #1 (Early pre-market discovery)
08:30  → Gap Scan #2 (Morning refresh — late-breaking movers)
09:00  → Portfolio Manager (Capital allocation plan)
09:00  → Pre-Market Pipeline (Aggregate, filter, AI analysis)
09:15  → Pre-Market Sanity Check (System health validation)
09:25  → Execution Bundle finalized and handed off
09:30  → MARKET OPEN — Pre-market phase ends
```

*HK Market follows a parallel timeline shifted to HKT (UTC+8).*

## 3. Cross-Cutting Functional Requirements

These requirements apply to *all* EPICs and features in the pre-market phase.

### FR-XC-01: Schedule Reliability
The pre-market phase must execute automatically on every US market trading day (Mon–Fri, excluding observed holidays) according to the fixed timeline. If any stage fails, the system must alert the operator and continue with degraded data rather than abort entirely.

### FR-XC-02: Auditability
Every decision, recommendation, and filter applied during pre-market must be persisted with a timestamp and trace identifier. An operator must be able to reconstruct exactly why a symbol was included, excluded, sized, or rejected.

### FR-XC-03: Determinism
Given identical inputs (same scan data, same portfolio state, same regime), the pre-market pipeline must produce identical outputs. Non-determinism (e.g., from LLM temperature) must be bounded and logged.

### FR-XC-04: Configurability
All thresholds (gap %, volume minimums, risk limits, position counts) must be externally configurable without code changes. Changes must take effect on the next trading day.

### FR-XC-05: Dual-Market Support
The pre-market phase must support both US equities (ET timezone) and HK equities (HKT timezone) with independent schedules, symbol universes, and market-specific rules.

### FR-XC-06: Graceful Degradation
If an external data source (e.g., gap scanner website) is unavailable, the system must fall back to alternative sources. If AI analysis is unavailable, rule-based analysis must still produce a valid execution bundle.

## 4. Output Artifacts (What Pre-Market Produces)

At the end of the pre-market phase, the system produces the following artifacts for handoff to market operations:

| Artifact | Description | Consumer |
|----------|-------------|----------|
| **Execution Bundle** | Canonical container with all gap candidates, AI recommendations, portfolio rebalancing orders, regime state, and trace context | Market Operations / Trade Execution |
| **Master Candidate List** | Consolidated, deduplicated list of gap candidates with rankings and scores | Operator review, audit |
| **AI Recommendations** | Per-symbol trading recommendations with confidence, sizing, stops, and targets | Position Sizer, Risk Manager |
| **Portfolio Recommendation** | Target portfolio allocation with rebalancing orders and cash reserve | Portfolio Manager |
| **Regime Report** | Current market regime classification with confidence and stability flag | Strategy selection, sizing adjustment |
| **Sanity Check Report** | Pass/fail status of all system health checks with remediation hints | Operator, monitoring |
| **Daily Readiness Report** | Executive summary of system health, scan results, and trading readiness | Operator |

### 4.1 Execution Bundle Content (Functional Specification)

The Execution Bundle is the single handoff artifact. It must contain:

1. **Scan Metadata**
   - Timestamp of pipeline completion
   - Date of intended trading session
   - Trace identifier for distributed tracing

2. **Gap Candidates** (0 to N)
   - Symbol, company name, exchange, sector, industry
   - Gap percentage and dollar amount
   - Pre-market price, previous close
   - Pre-market volume and relative volume vs. 30-day average
   - Float category and market cap category
   - ATR, RSI, SMA50 where available
   - News catalyst classification and sentiment score
   - Data completeness score (0–100%)

3. **AI Recommendations** (0 to N)
   - Symbol
   - Recommendation grade: STRONG_BUY, BUY, CAUTIOUS, WATCH, AVOID
   - Confidence level (0.0–1.0)
   - Total score (0–100) with breakdown across:
     - Gap quality (0–25)
     - Technical setup (0–20)
     - Sentiment (0–20)
     - Liquidity (0–15)
     - Risk (0–10)
     - Market context (0–10)
   - Suggested position size (% of portfolio)
   - Suggested entry, stop, and target prices
   - Risk-reward ratio
   - Reasoning narrative
   - Key decision factors and risk factors

4. **Portfolio Recommendation**
   - Target positions: symbol → target dollar amount
   - Rebalancing orders (BUY/SELL/TRIM per symbol)
   - Cash target (% of NAV to hold in reserve)
   - Overall confidence and reasoning

5. **Regime State**
   - Detected regime: LOW_VOL, MID_VOL, HIGH_VOL, or BEARISH
   - Regime confidence (0.0–1.0)
   - Stability flag (has regime persisted for minimum bars)

6. **Safety State**
   - Kill switch status: ACTIVE / INACTIVE
   - Circuit breaker status: HALTED / REDUCED / NORMAL
   - Pre-trade integrity summary

## 5. Functional Validation Matrix

This matrix defines how to validate that the pre-market phase is functioning correctly.

| Validation ID | Description | Expected Output | Pass Criteria |
|---------------|-------------|-----------------|---------------|
| FV-001 | Run Gap Scan #1 at 04:30 ET | JSON file with candidate list | ≥ 5 candidates with complete fields (symbol, gap%, volume, price) |
| FV-002 | Run Gap Scan #2 at 08:30 ET | JSON file with candidate list | ≥ 5 candidates; overlaps with Scan #1 should show trend direction (rising/fading) |
| FV-003 | Run Pre-Market Pipeline at 09:00 ET | Master candidate list + AI recommendations | ≤ 50 master candidates after deduplication; ≥ 3 AI recommendations with scores ≥ 60 |
| FV-004 | Run Portfolio Manager | Portfolio recommendation with allocations | Cash reserve ≥ 15% of NAV; no single position > 15% of NAV; max 10 positions |
| FV-005 | Run Position Sizer for a STRONG_BUY recommendation | Position size result | Shares > 0; risk % ≤ 2% of account equity; dollar amount ≤ max position limit |
| FV-006 | Run Regime Detection | Regime prediction | Regime is one of LOW_VOL / MID_VOL / HIGH_VOL / BEARISH; confidence ≥ 0.6 |
| FV-007 | Run Pre-Market Sanity Check | Sanity report with 20+ checks | All checks PASS; any FAIL triggers CRITICAL alert with remediation hint |
| FV-008 | Generate Execution Bundle | Bundle file | Contains all 6 sections; all symbols are valid ticker formats; trace ID present |
| FV-009 | Verify blacklist filtering | Blacklisted symbols excluded | No blacklisted symbol appears in master candidate list |
| FV-010 | Verify trend-down filtering | Fading gaps excluded | Symbols with gap decay > trend threshold excluded from master list |

## 6. Glossary

| Term | Definition |
|------|------------|
| **Gap %** | Percentage difference between previous closing price and current pre-market price. |
| **Relative Volume** | Pre-market volume divided by the 30-day average daily volume. Values > 2.0 indicate unusual interest. |
| **Float** | Number of shares available for public trading. Low float (< 20M) can lead to exaggerated moves. |
| **ATR** | Average True Range — a volatility measure used for stop-loss placement. |
| **Regime** | Classified market state (LOW_VOL, MID_VOL, HIGH_VOL, BEARISH) that affects strategy and sizing. |
| **Portfolio Heat** | Total risk exposure as a percentage of account equity across all open positions. |
| **Kelly Fraction** | Optimal position size fraction derived from win rate and payoff ratio. |
| **Execution Bundle** | The canonical handoff artifact containing all pre-market outputs for market operations. |
| **Rebalancing Order** | A BUY, SELL, or TRIM instruction generated when the current portfolio deviates from the target. |
| **Catalyst** | The news event or fundamental driver causing a price gap (e.g., earnings, FDA approval, M&A). |
