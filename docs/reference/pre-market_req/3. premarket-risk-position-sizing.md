# EPIC-3: Risk Management & Position Sizing

> **EPIC ID:** EPIC-3  
> **Status:** Active  
> **Phase:** Pre-Market (09:00–09:25 ET, during pipeline execution)  
> **Depends On:** EPIC-2 (AI Recommendations)  
> **Out of Scope:** Intraday risk monitoring, live order risk checks (post-market-open), broker-specific margin calculations

---

## 1. EPIC Description

As a **trading system operator**, I want every recommended position to be sized according to account equity, stop distance, volatility regime, and portfolio risk limits, so that no single trade can cause catastrophic loss and the portfolio as a whole remains within safe heat boundaries.

### 1.1 Business Value

- **Capital Preservation:** Proper position sizing ensures that even a string of losing trades cannot destroy the account.
- **Optimal Growth:** The Kelly criterion component sizes positions to maximize long-term geometric growth rate without undue risk.
- **Regime Adaptation:** Position sizes are automatically reduced in high-volatility regimes and increased (within limits) in low-volatility regimes.
- **Portfolio-Level Safety:** Portfolio heat monitoring prevents over-concentration and correlation risk.

### 1.2 Functional Boundaries

```
IN SCOPE:
  ✓ Position size calculation per trade signal
  ✓ Monte Carlo simulation for risk estimation
  ✓ Kelly criterion for optimal fraction
  ✓ Volatility regime adjustments
  ✓ Pre-trade risk validation (before orders are constructed)
  ✓ Portfolio heat monitoring and limits
  ✓ Pre-trade integrity checks (stale orders, reconciliation)
  ✓ Circuit breaker evaluation

OUT OF SCOPE:
  ✗ Live order submission or fill validation
  ✗ Intraday position monitoring (every-15-min audit)
  ✗ Broker margin requirement calculations
  ✗ Post-trade P&L attribution
  ✗ Tax or regulatory reporting
```

## 2. Features

### Feature-3.1: Monte Carlo Position Sizer

#### Description
A position sizing engine that uses Monte Carlo simulation of trade outcomes and the Kelly criterion to calculate the optimal number of shares for each trade signal. It respects stop distance, account equity, volatility regime, and configured risk limits.

#### User Stories

##### US-3.1.1: Optimal Position Size Calculation
> **As a** trading system,  
> **I want** to calculate the optimal number of shares for each trade signal based on account equity, entry price, stop price, and signal strength,  
> **So that** each position risks no more than a defined percentage of equity while maximizing expected growth.

**Acceptance Criteria:**
- [ ] Position size is calculated using the formula: `shares = risk_amount / stop_distance`.
- [ ] `risk_amount = account_equity × base_risk_pct × signal_strength × kelly_fraction × regime_multiplier`.
- [ ] Base risk percentage is configurable (default: 1.0% of equity per trade).
- [ ] Signal strength scales risk linearly from 0% to 100% of base risk (range: 0.0–1.0).
- [ ] Kelly fraction is derived from historical win rate and payoff ratio, capped at half-Kelly for safety (default cap: 0.25).
- [ ] Regime multipliers are: LOW_VOL = 1.25×, MID_VOL = 1.0×, HIGH_VOL = 0.60×.
- [ ] The result includes: shares, dollar amount, risk amount, risk percentage, Kelly fraction, confidence, and reasoning.
- [ ] If the calculated shares are zero, the system logs the root cause (e.g., "stop_distance too large", "signal_strength near zero").

**Functional Validation:**
```
Input:  symbol="AAPL", entry_price=$150.00, stop_price=$145.00,
        account_equity=$100,000, signal_strength=0.75,
        regime="low", historical_returns=[0.02, -0.01, 0.03, ...]
Output: PositionSizeResult

Validation:
  1. stop_distance = $150 - $145 = $5.00.
  2. base_risk = $100,000 × 1% = $1,000.
  3. signal_adjusted_risk = $1,000 × 0.75 = $750.
  4. regime_multiplier = 1.25 (low vol).
  5. kelly_fraction ≤ 0.25.
  6. risk_amount = $750 × kelly_fraction × 1.25.
  7. shares = int(risk_amount / $5.00).
  8. dollar_amount = shares × $150.
  9. risk_pct = risk_amount / $100,000.
  10. risk_pct ≤ 2.0% (hard ceiling).
  11. shares > 0 for valid inputs.
```

##### US-3.1.2: Risk-Reward Gating
> **As a** risk manager,  
> **I want** to block trades where the risk-reward ratio is below a minimum threshold,  
> **So that** the system only takes trades with favorable payoff profiles.

**Acceptance Criteria:**
- [ ] The minimum risk-reward threshold is configurable (default: 1.5:1).
- [ ] If target_price is provided and `(target_price - entry_price) / (entry_price - stop_price) < min_rr`, the trade is blocked.
- [ ] Blocked trades return a PositionSizeResult with `shares = 0` and `reasoning` explaining the R/R violation.
- [ ] The minimum R/R threshold may be adjusted per regime (e.g., higher in HIGH_VOL).

**Functional Validation:**
```
Input:  entry=$150, stop=$145, target=$152, min_rr=1.5
Output: Blocked (shares=0)

Validation:
  1. R/R = (152 - 150) / (150 - 145) = 2 / 5 = 0.4.
  2. 0.4 < 1.5 → trade blocked.
  3. Reasoning contains "risk-reward ratio 0.4 below minimum 1.5".

Input:  entry=$150, stop=$145, target=$160, min_rr=1.5
Output: Allowed

Validation:
  1. R/R = (160 - 150) / (150 - 145) = 10 / 5 = 2.0.
  2. 2.0 ≥ 1.5 → trade allowed.
```

##### US-3.1.3: Monte Carlo Risk Estimation
> **As a** risk manager,  
> **I want** to simulate future trade outcomes using Monte Carlo methods,  
> **So that** position sizes account for tail risk and drawdown potential.

**Acceptance Criteria:**
- [ ] When historical returns are available, the sizer runs a Monte Carlo simulation with configurable iterations (default: 10,000).
- [ ] The simulation produces: expected return, volatility, max drawdown, Sharpe ratio, 95% VaR, and optimal Kelly fraction.
- [ ] If Monte Carlo exceeds a time limit (default: 100ms), the sizer falls back to Kelly-only calculation.
- [ ] Monte Carlo results are cached per (returns_hash, regime) tuple to avoid recomputation.
- [ ] The simulation uses only historical returns provided; it does not fetch new data.

**Functional Validation:**
```
Input:  historical_returns = 90 days of daily returns,
        mc_iterations = 10,000
Output: MonteCarloResult

Validation:
  1. expected_return is a finite number.
  2. volatility > 0.
  3. max_drawdown ≤ 0 (negative or zero).
  4. sharpe_ratio is a finite number.
  5. var_95 is negative (representing loss).
  6. kelly_fraction is between -1.0 and 1.0.
  7. If called twice with same returns and regime, second call uses cache
     (response time < 1ms vs > 50ms for first call).
```

### Feature-3.2: Pre-Trade Risk Manager

#### Description
A risk validation layer that checks every proposed trade against portfolio-level constraints before it is approved for execution. It enforces position limits, sector limits, correlation limits, and circuit breaker rules.

#### User Stories

##### US-3.2.1: Trade Risk Validation
> **As a** trading system,  
> **I want** to validate every proposed trade against portfolio risk rules before approving it,  
> **So that** no trade violates hard limits on exposure, concentration, or correlation.

**Acceptance Criteria:**
- [ ] The risk manager checks:
  1. **Kill Switch:** Trading must be allowed (kill switch inactive).
  2. **Circuit Breaker:** If circuit breaker is HALTED, all trades are blocked. If REDUCED_50, position sizes are halved.
  3. **Position Limit:** New position value + existing position value ≤ max_position_pct of NAV (default: 15%).
  4. **Sector Limit:** New sector value ≤ max_sector_pct of NAV (default: 25% for US, 20% for HK).
  5. **Correlation Limit:** If the new symbol's correlation with any existing position exceeds the threshold (default: 0.85), risk level is elevated.
  6. **Portfolio Heat:** New heat (existing heat + new trade risk) ≤ max_heat (default: 50% of NAV).
  7. **Daily PnL Halt:** If daily PnL ≤ daily_pnl_halt threshold (default: -3%), all new entries are blocked.
- [ ] Each check produces a pass/fail result with a reason string.
- [ ] The overall result is ALLOWED only if all checks pass.
- [ ] Warnings are collected for checks that pass but are near thresholds.

**Functional Validation:**
```
Input:  Proposed trade: AAPL, $15,000; Existing: AAPL=$5,000; NAV=$100K;
        max_position_pct=15%; current_heat=40%; new_risk=2%
Output: RiskCheckResult

Validation:
  1. Total AAPL position would be $20K = 20% of NAV > 15% → BLOCKED.
  2. Reason = "Position limit exceeded: AAPL would be 20.0% of NAV".

Input:  Proposed trade: MSFT, $5,000; Existing heat=40%; new_risk=2%;
        max_heat=50%; daily_pnl=-1%
Output: ALLOWED

Validation:
  1. New heat = 42% ≤ 50% → pass.
  2. daily_pnl = -1% > -3% → pass.
  3. All other checks pass → ALLOWED.
```

##### US-3.2.2: Circuit Breaker Management
> **As a** risk manager,  
> **I want** automatic circuit breakers that halt or reduce trading when daily or weekly P&L crosses critical thresholds,  
> **So that** drawdowns are contained before they become catastrophic.

**Acceptance Criteria:**
- [ ] Circuit breaker states:
  - **NORMAL:** All trading permitted.
  - **REDUCED_50:** New position sizes reduced by 50%; rebalancing still permitted.
  - **HALTED:** No new entries permitted; exits still permitted.
- [ ] Triggers:
  - REDUCED_50: Daily PnL ≤ daily_pnl_reduce threshold (default: -2% for US, -1.5% for HK).
  - HALTED: Daily PnL ≤ daily_pnl_halt threshold (default: -3% for US, -2.5% for HK).
- [ ] Circuit breaker status is evaluated before every trade approval.
- [ ] Status transitions are logged with timestamp, trigger value, and threshold.
- [ ] The circuit breaker auto-resets at the start of the next trading day.

**Functional Validation:**
```
Input:  Daily PnL = -2.5%; daily_pnl_reduce = -2.0%; daily_pnl_halt = -3.0%
Output: Circuit breaker = REDUCED_50

Validation:
  1. -2.5% ≤ -2.0% → REDUCED_50 triggered.
  2. -2.5% > -3.0% → HALTED not triggered.
  3. New position sizes are halved.

Input:  Daily PnL = -3.5%; daily_pnl_halt = -3.0%
Output: Circuit breaker = HALTED

Validation:
  1. -3.5% ≤ -3.0% → HALTED triggered.
  2. All new entries blocked.
  3. Exit orders still permitted.
```

### Feature-3.3: Pre-Trade Integrity Checker

#### Description
A multi-layer integrity verification system that runs before any new orders are placed. It detects stale pending orders, reconciles local state with broker state, flags position discrepancies, and validates kill switch sanity.

#### User Stories

##### US-3.3.1: Stale Order Detection and Cleanup
> **As a** trading system,  
> **I want** to detect and auto-cancel stale pending orders from previous sessions,  
> **So that** the system does not enter the trading day with ambiguous order state.

**Acceptance Criteria:**
- [ ] Orders in PENDING_NEW status older than a configurable threshold (default: 4 hours) are flagged as stale.
- [ ] Stale orders are automatically cancelled.
- [ ] Each cancellation is logged with order ID, symbol, age, and reason.
- [ ] The cancellation count is included in the integrity check summary.

**Functional Validation:**
```
Input:  Order #123: PENDING_NEW, created 5 hours ago
        Order #456: PENDING_NEW, created 2 hours ago
Output: Order #123 cancelled; Order #456 retained

Validation:
  1. Order #123 age = 5h > 4h threshold → cancelled.
  2. Order #456 age = 2h ≤ 4h → retained.
  3. Log shows: "Cancelled stale order #123 (age: 5h)".
```

##### US-3.3.2: Broker Reconciliation
> **As a** trading system,  
> **I want** to reconcile local order state with the broker's open orders,  
> **So that** orders that were filled or cancelled externally are reflected locally.

**Acceptance Criteria:**
- [ ] The system queries the broker for all open orders.
- [ ] For each local PENDING_NEW order, if it does not exist in the broker's open orders, it is marked as CANCELLED locally.
- [ ] Discrepancies are logged with the local order ID and the discrepancy type.
- [ ] The reconciliation count is included in the integrity check summary.

**Functional Validation:**
```
Input:  Local orders: #100 (PENDING_NEW), #101 (PENDING_NEW)
        Broker open orders: #100
Output: Order #101 marked CANCELLED locally

Validation:
  1. #100 exists at broker → no action.
  2. #101 missing at broker → marked CANCELLED.
  3. Log: "Reconciled order #101: broker has no record, marked CANCELLED".
```

##### US-3.3.3: Position Discrepancy Detection
> **As a** risk manager,  
> **I want** to detect discrepancies between the local position ledger and the broker's position snapshots,  
> **So that** trades are not placed based on stale position data.

**Acceptance Criteria:**
- [ ] The system compares local position records against broker-reported positions.
- [ ] Discrepancies are flagged when:
  - A position is CLOSED locally but OPEN at the broker.
  - A position is OPEN locally but missing at the broker.
  - Position quantities differ by more than a rounding tolerance.
- [ ] Discrepancies are logged at WARNING level with full details.
- [ ] The discrepancy count is included in the integrity check summary.

**Functional Validation:**
```
Input:  Local: AAPL qty=100; Broker: AAPL qty=0
Output: Discrepancy flagged

Validation:
  1. Local shows AAPL open, broker shows closed → discrepancy.
  2. Log: "Position discrepancy: AAPL local=100, broker=0".
```

##### US-3.3.4: Stale Day-Position Detection
> **As a** risk manager,  
> **I want** to be alerted when day-trade positions have been open for more than 3 market days,  
> **So that** unintended swing positions are flagged for operator review.

**Acceptance Criteria:**
- [ ] Positions with timeframe "day" are checked for age in market days (Mon–Fri only).
- [ ] If a day position has been open for > 3 market days, an ALERT is raised.
- [ ] The alert includes: symbol, entry date, days open, and suggested action (close position).
- [ ] This is an alert-only check; positions are NOT auto-closed.

**Functional Validation:**
```
Input:  Position: TSLA, timeframe="day", opened on Monday, today is Friday
Output: ALERT raised

Validation:
  1. Market days open: Mon→Tue→Wed→Thu→Fri = 4 market days (if counting inclusively)
     or 3 full market days. Alert triggers when > 3.
  2. Alert includes symbol, open date, and days count.
  3. Position is NOT automatically closed.
```

### Feature-3.4: Portfolio Heat Monitor

#### Description
A real-time tracker of total portfolio risk exposure, expressed as a percentage of account equity. Portfolio heat is the sum of individual position risk amounts (entry price – stop price) × shares, divided by account equity.

#### User Stories

##### US-3.4.1: Portfolio Heat Calculation
> **As a** risk manager,  
> **I want** to calculate the total portfolio heat before market open,  
> **So that** I know how much of the account is currently at risk.

**Acceptance Criteria:**
- [ ] Portfolio heat = Σ(position_risk_amount) / account_equity.
- [ ] Position risk amount = |entry_price – stop_price| × shares.
- [ ] Heat is calculated for all open positions plus proposed new positions.
- [ ] If heat exceeds the configured maximum (default: 50% of NAV), new entries are blocked.
- [ ] Heat history is persisted for trend analysis.

**Functional Validation:**
```
Input:  Account equity = $100K
        Position A: entry=$100, stop=$95, shares=100 → risk=$500
        Position B: entry=$50, stop=$48, shares=200 → risk=$400
        Proposed C: entry=$150, stop=$145, shares=100 → risk=$500
Output: Current heat = ($500 + $400) / $100K = 0.9%
        Proposed total heat = ($500 + $400 + $500) / $100K = 1.4%

Validation:
  1. Current heat correctly sums existing positions.
  2. Proposed heat includes new position.
  3. If max_heat = 50%, proposed heat of 1.4% is well within limit.
```

## 3. Functional Requirements

### FR-3.1: Sizing Non-Negativity
The position sizer must never return negative shares, negative dollar amounts, or negative risk amounts. Invalid inputs must produce shares=0 with an explanatory reasoning.

### FR-3.2: Hard Risk Ceiling
No single trade may risk more than 2% of account equity, regardless of signal strength, Kelly fraction, or regime multiplier. This is an absolute ceiling that overrides all other calculations.

### FR-3.3: Circuit Breaker Override
When the circuit breaker is HALTED, no new entry signals may be approved. When REDUCED_50, all approved position sizes are halved before submission.

### FR-3.4: Integrity Check Timing
Pre-trade integrity checks must complete within 60 seconds. If they exceed this time, the system must abort the trading run with a CRITICAL alert.

### FR-3.5: Heat Persistence
Portfolio heat must be calculated and persisted at least once per trading day before market open. Heat history must be retained for 90 days.

### FR-3.6: Venue-Specific Limits
Risk limits may differ by market venue (US vs. HK). The risk manager must apply the correct venue limits based on the symbol's market.
