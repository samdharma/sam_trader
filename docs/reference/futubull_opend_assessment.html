1.2 Tick-Level Data for Algo Trading

OpenD explicitly supports tick-by-tick subscriptions via SubType.TICKER. Data is pushed asynchronously to callbacks after subscription. You can also pull the latest buffered tick data on-demand via get_rt_ticker without additional frequency limits (reads from OpenD's local cache).

Gating factors:
  • Subscription Quotas — Concurrent real-time subscriptions are capped by account tier (see table below).
  • Frequency Limits — Request-based interfaces have rate limits (e.g., market snapshot max 60 req/30s). Subscription push data has no frequency limit.
  • 1-Minute Lock — You cannot unsubscribe and release quota until at least 1 minute has passed.

Each stock + data type combo consumes 1 quota. For 50 US symbols with quote + tick + order book, that's 150 subscriptions.

Assessment: Tick-level data is viable for algo. For a focused strategy running <100 instruments, the free tier (with modest assets) is sufficient. Scale beyond that requires higher tiers.

5. NautilusTrader Adapter Landscape

5.1 Community Adapter: nautilus-futu

A mature third-party adapter (loadstarCN/nautilus-futu) exists with these characteristics:

  • Rust + PyO3 core — Handles TCP protobuf protocol, AES encryption
  • No futu-api dependency — Uses Rust prost to avoid Python protobuf conflicts
  • Full coverage: Data client, execution client, instrument provider
  • Market support: HK, US, A-share
  • Config pattern: FutuDataClientConfig / FutuExecClientConfig — mirrors our IB config pattern

5.2 Build vs Adopt Decision

PathEffortRiskRecommendation
Use nautilus-futu package Low (~1–2 weeks integration) Third-party maintenance, version drift Fastest, but adds external dep
Build custom adapter High (~2–3 months) Full ownership, longer time-to-market Full control, aligns with "reduce deps" goal
Hybrid (recommended) Medium (~4–6 weeks) Low — proven patterns, our own code Adopt architecture, build implementation

7. Recommendations

  1. Target US equities first — Free LV2 + deeper order book + clearer symbology. HK can follow.
  2. Paper trade via OpenD first — trd_env=0 (simulate) requires no unlock password. Verify tick quality and latency before going live.
  3. Plan for auth persistence in Docker — Copy pre-verified OpenD config/state into a volume-mounted directory to avoid SMS loops on restart.
  4. Architect subscription management carefully — Given quota limits, build a subscription manager that prioritizes active instruments and releases unused ones (respecting the 1-minute minimum).
  5. Futu AI Skills: optional — Install for research/anomaly detection if they fit the pre-market pipeline, but don't block adapter work on them.
