1. Overview

SAM Trader V3 is a production-grade autonomous trading platform built on NautilusTrader v1.227+. It runs inside Docker on macOS (or Linux), trading US and HK equities through Futu OpenD (primary broker) and Interactive Brokers (secondary, optional). The system is deployed in Hong Kong and operates on HKT (GMT+8).

1.1 What This Guide Covers

This is the complete reference for the human operator. It covers:

  • New installation and first-run setup
  • Timezone-aware daily schedule across US and HK market sessions
  • Pre-market pipeline: gap scanning, AI scoring, risk checks, bundle generation
  • Human review gates — where the operator inspects and approves before each session
  • Market-hours monitoring via dashboard and CLI
  • Post-market review and wrap-up
  • Updates, hot-fixes, and rollback procedures
  • Broker-specific operating guides (Futu OpenD + IB Gateway)
  • Bundle management (YAML registry, versioning, validation)
  • Maintenance: backup, restore, log rotation, volume management
  • Incident response: kill switch, halt, circuit breakers
  • Full sam CLI reference and troubleshooting

4.4 Step-by-Step Installation

⚠ BEFORE YOU START: Safety Check This installation uses your real Futubull credentials. The system defaults to SIMULATE (paper trading), but one misconfiguration could route orders to your live account. Read every step. Keep the Futubull app open to the Paper Trading tab during testing.

cd ~/Trading

cd sam_trader

Step 3: Generate Your MD5 Password Hash

⚠ The -n flag is essential. Without it, echo adds a trailing newline and the hash will be wrong — login will fail.

Method A — macOS built-in md5:

echo -n 'YourFutuLoginPassword' | md5
# Example output: 5f4dcc3b5aa765d61d8327deb882cf99

Method B — OpenSSL (always available):

printf '%s' 'YourFutuLoginPassword' | openssl md5
# Example output: (stdin)= 5f4dcc3b5aa765d61d8327deb882cf99

Method C — Python (most reliable, cross-platform):

python3 -c "import hashlib; print(hashlib.md5('YourFutuLoginPassword'.encode()).hexdigest())"
# Example output: 5f4dcc3b5aa765d61d8327deb882cf99

Copy the 32-character hex string into FUTU_ACCOUNT_PWD_MD5= in your .env.

Step 4: Generate the RSA Key (Futu Required)

Futu OpenD binds to 0.0.0.0 inside Docker so other containers can reach it over sam-net. Futu requires RSA encryption on the trading interface when listening on all interfaces. Without this key, market data works but order submission fails with an encryption error.

ssh-keygen -t rsa -b 1024 -m PEM -f docker/futu-opend/futu.pem -N ""

# Verify:
ls -l docker/futu-opend/futu.pem
# Expected: -rw-------  1 you  staff  887 ... docker/futu-opend/futu.pem
⚠ Never commit futu.pem. *.pem is already in .gitignore. The key is mounted into both sam-futu-opend and sam-trader containers automatically at /.futu/futu.pem.

Step 8: Start Futu OpenD

This step is where first-time authentication happens. The Futu OpenD container downloads the binary on first start (~405 MB, 1–3 minutes depending on connection). Subsequent restarts use the cached binary from the persistent volume.

docker compose -f docker/docker-compose.yml --env-file .env --profile futu up -d sam-futu-opend

# Watch the logs in real time:
docker logs -f sam-futu-opend

What you will see (first start):

  1. Binary download: Downloading Futu OpenD ... → wait 1–3 minutes
  2. Extraction: Extracting Futu OpenD ...
  3. Startup: Futu OpenD ready: /home/futu/.com.futunn.FutuOpenD/.../FutuOpenD
  4. Login attempt: One of the scenarios below

Step 9: SMS Verification (If Needed)

If Futu requires SMS verification, you need to send the code to OpenD's telnet console on port 22222. This must be done by you — the code comes to your phone.

Method A: Python helper (no install needed):

# Replace 123456 with the actual SMS code from your phone:
python3 -c "
import socket, time, select
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(('localhost', 22222))
s.setblocking(0)
time.sleep(0.5)
ready, _, _ = select.select([s], [], [], 2)
if ready: print(ready[0].recv(1024).decode('utf-8', errors='replace').strip())
cmd = 'input_phone_verify_code -code=123456\n'.encode()
s.send(cmd)
time.sleep(2)
ready, _, _ = select.select([s], [], [], 3)
if ready: print(ready[0].recv(1024).decode('utf-8', errors='replace').strip())
s.close()
"

Method B: Telnet (install via brew install telnet):

/opt/homebrew/bin/telnet localhost 22222
# Type: input_phone_verify_code -code=123456
# Press Enter

After sending the code, check the logs:

docker logs sam-futu-opend --tail 10
# Should show: [OpenD] Login success
#               [OpenD] Connect to Futu server success
✅ One-Time Setup Complete The login session is cached in the futu_opend_data Docker volume. Future restarts skip SMS and questionnaire. Only docker compose down -v (wiping volumes) requires re-authentication.

10.2 Graceful Restart Flow

The restart process preserves strategy/actor state via Redis:

  1. save_state() — all actors and strategies serialize state to Redis
  2. stop() — TradingNode stops, connections closed
  3. build(new_config) — new TradingNode built from updated config
  4. run() — new node starts
  5. State restored — strategies/actors reload state from Redis

Total downtime: ~5–10 seconds. Futu OpenD and IB Gateway containers are unaffected (they run independently). Subscriptions are restored automatically on reconnect.
